1. System Overview

The tree evaluation system determines what a player should bid given their hand and the auction so far. It replaces a flat rule-list architecture with a layered decision model that separates what happened in the auction (protocol), what the hand looks like (hand tree), what the hand wants to say (intent), what the partnership context means (dialogue state), and how interference modifies behavior (overlays).

Why this architecture? A flat condition-list system had 11 known gaps. Most critically: (1) interference blindness — all rules assumed uncontested auctions; (2) no negative inference — flat condition lists can't express "this convention path was rejected, so these hand constraints DON'T apply." Rule trees were chosen because tree path rejection data is the only architecture that enables structural negative inference.

2. Node Types

Every tree is built from exactly three node types. The union RuleNode = DecisionNode | IntentNode | FallbackNode is exhaustively matched everywhere with TypeScript's never guard.

DecisionNode

Binary branching question. Tests one condition, follows YES or NO child.

{
  type: "decision",
  name: "has-4-hearts",
  condition: suitMin(1, "hearts", 4),
  yes: /* RuleNode */,
  no:  /* RuleNode */,
  metadata?: DecisionMetadata
}
Subtypes: HandDecisionNode restricts condition to HandCondition and children to HandNode.
IntentNode

Terminal leaf. Carries semantic intent + default concrete call. The only leaf type.

{
  type: "intent",
  nodeId: "intent-7",  // auto-assigned
  name: "stayman-ask",
  meaning: "Asks for a 4-card major",
  intent: {
    type: AskForMajor,
    params: {}
  },
  defaultCall: (ctx) => bid(2, Clubs),
  metadata?: BidMetadata,
  alert?: BidAlert
}
nodeId: Auto-assigned "intent-N" via monotonic counter in intentBid(). Used for identity matching throughout the pipeline (survives spread copies).
FallbackNode

Terminal "no match" leaf. Convention doesn't apply to this hand/auction path.

{
  type: "fallback",
  reason?: "too-weak"
}
Note: SAYC uses saycPass() at terminal positions since it's the catch-all convention.

Actor-Aware Conditions (Phase 6)

partnerBidMade() and opponentBidMade() use areSamePartnership() to determine actor relationship relative to the evaluating seat. These replaced seat-agnostic bidMade() in protocol triggers where actor identity matters (e.g., Stayman triggers now use partnerBidMade(1, NT) to ensure the evaluating seat's partner opened 1NT).

Inference metadata: Each HandCondition carries an optional .inference field (e.g., { type: "hcp-min", params: { min: 8 } }) that the inference engine uses to narrow beliefs about what a player's hand contains when they make a bid. AuctionConditions must NOT carry inference — enforced by test. negatable defaults true; set false for isBalanced() because NOT balanced does not cleanly mean unbalanced (could be semi-balanced).

4. Tree Evaluator

evaluateTree() in core/tree-evaluator.ts is the heart of hand evaluation. It walks a RuleNode tree iteratively (stack-safe for deep trees), following YES/NO branches based on condition results, until it reaches a leaf.

TreeEvalResult

FieldTypePurpose
matchedIntentNode | nullThe terminal leaf reached, or null if fallback

Why track rejections? When a player bids 2C Stayman, the inference engine knows they passed "8+ HCP? YES" and "4-card major? YES". But equally important: if they had bid 2NT instead, we'd know they passed "8+ HCP? YES" but were rejected at "4-card major? NO". The rejectedDecisions list powers negative inference — narrowing what a hand doesn't have.

Iterative Traversal (Not Recursive)

The evaluator uses an iterative for(;;) loop with a switch statement, not recursion. This is stack-safe for arbitrarily deep trees (Stayman's rebid tree is 10+ levels deep). The default case uses TypeScript's never exhaustiveness check to catch unhandled node types at compile time.

Protocol Evaluation Algorithm

evaluateProtocol() walks rounds sequentially using a cursor that advances by 2 for each matched round (one conversation turn = one partnership bid + opponent response slot). It now accepts optional triggerOverrides from overlay patches.

function evaluateProtocol<T>(
  proto: ConventionProtocol<T>,
  context: BiddingContext,
  triggerOverrides?: ReadonlyMap<string, readonly SemanticTrigger[]>,
): ProtocolEvalResult<T>

// Simplified logic:
let cursor = 0;
let established = { role: computeRole(entries, seat) };
let activeRound = null;

for (const round of protocol.rounds) {
  const windowedCtx = contextAtCursor(context, cursor);

  // Use overlay trigger overrides if provided
  const triggers = triggerOverrides?.get(round.name) ?? round.triggers;

  const trigger = triggers.find(t => t.condition.test(windowedCtx));
  if (!trigger) break;  // Auction hasn't reached this round

  cursor += 2;  // Advance to next partnership turn
  established = { ...established, ...trigger.establishes };

  // seatFilter: evaluated against FULL context (not windowed!)
  if (!round.seatFilter || round.seatFilter.test(context)) {
    activeRound = round;  // This seat acts in this round
  }
}

const handTree = resolveHandTree(activeRound.handTree, established);
return evaluateTree(handTree, context);

Key Concepts

Cursor Windowing

Trigger conditions see a windowed view of the auction — entries.slice(0, cursor + 2). This lets Round 2's trigger see "2C was bid" without seeing Round 3's auction entries. The cursor advances by 2 because partnerships are always 2 positions apart in the clockwise auction (N-E-S-W).

seatFilter vs Triggers

Triggers detect what happened (milestones). seatFilter determines who acts. If seatFilter fails, the cursor STILL advances (the milestone happened) but activeRound is NOT updated. This separation is crucial for multi-seat protocols.

Dynamic handTree

When handTree is a function, it receives established (accumulated trigger context). Stayman Round 3 uses this to dispatch to different subtrees based on showed (which major was shown) and openingLevel (1NT vs 2NT). If it throws, the protocol evaluator catches and returns no match.

computeRole()

Derives AuctionRole (opener/responder/competitive/rebidder) from auction entries. Opener = first non-pass bidder. Responder = partner of opener. Competitive = opponent's partnership. Rebidder = opener who has bid more than once.

Two-Pass Transition Model

All 4 conventions use two-pass mode. For each auction entry, convention-specific rules fire first (first match wins, full apply). Then baseline rules fire separately (first match wins, backfill only — convention values always take precedence). This ensures conventions can override baseline behavior while still getting interference detection, NT opening detection, etc. for free.

Replay, not incremental: DialogueState is rebuilt from scratch for each evaluation. There's no mutable state being updated bid by bid. This makes the system deterministic and eliminates state synchronization bugs. The cost is O(n) per evaluation where n = auction length, but bridge auctions are typically under 20 entries.
Causality contract (Phase 6): TransitionRule.matches() and effects() receive an optional entryIndex (0-based). Rules MUST NOT examine auction entries beyond this index. computeDialogueState() passes the current loop index. This prevents rules from "seeing the future" and ensures deterministic replay.

Resolver Pipeline

// Discriminated union — do NOT use null to mean "declined"
type ResolverResult =
  | { status: "resolved"; calls: readonly ResolvedIntent[] }  // Multi-encoding!
  | { status: "use_default" }
  | { status: "declined" };  // Excludes candidate entirely

type IntentResolverFn = (
  intent: SemanticIntent,
  state: DialogueState,
  context: BiddingContext,
) => ResolverResult;

// Resolvers are plain data on ConventionConfig:
readonly intentResolvers?: ReadonlyMap<string, IntentResolverFn>;
Why separate intent from call? A "show held suit" intent maps to 2H after 1NT Stayman, but 3H after 2NT Stayman. The intent is the same; the concrete call varies by context. Resolvers compute the right level based on DialogueState (e.g., staymanLevel(state) derives level from state.familyId). SAYC uses empty resolvers — all calls are deterministic via defaultCall.

Multi-Encoding Resolvers (Phase 5)

A resolver can return multiple ResolvedIntent alternatives (the calls array). The candidate generator tries each encoding in order and picks the first legal one. If none are legal, it uses the first encoding (marked illegal). This handles cases where the "natural" encoding of an intent is already taken by a prior bid, so an alternative encoding must be used.

"declined" vs null: A resolver returning { status: "declined" } excludes the candidate entirely — the hand can't express this intent in the current context. A Map miss (no resolver registered) returns null from resolveIntent(), which falls back to defaultCall. This distinction prevents accidental fallback when a convention explicitly decides an intent is inexpressible.

8. Overlay System

Overlays handle interference and competition. When opponents double or overcall, the convention's normal tree may not apply. Rather than adding interference branches to every tree, overlays patch the behavior for specific protocol rounds based on dialogue state. Multiple overlays can compose on the same round.

ConventionOverlayPatch Structure

interface ConventionOverlayPatch {
  id: string;                // Unique identifier
  roundName: string;         // Must match a ProtocolRound.name
  priority?: number;         // Lower = higher precedence (default 0)
  matches(state: DialogueState): boolean;  // Activation predicate

  // Five optional hooks:
  replacementTree?: HandNode;                    // Full tree swap
  suppressIntent?(intent, ctx): boolean;         // Remove candidates
  addIntents?(ctx): CollectedIntent[];           // Inject candidates
  overrideResolver?(intent, ctx): Call | null;   // Override resolution
  triggerOverrides?: ReadonlyMap<string, readonly SemanticTrigger[]>;
}

Overlay Resolution

buildEffectiveContext() resolves active overlays: iterates config overlays, checks roundName matches the active protocol round, and calls matches() with the computed dialogue state. All matching overlays are collected and sorted by priority (lower number = higher precedence, default 0).

Trigger Overrides (Phase 5)

Overlays can override protocol trigger conditions via triggerOverrides. collectTriggerOverrides() filters matching overlays, sorts by priority, and merges per round-name with first-wins semantics. Used in evaluateBiddingRules() to adjust how rounds are detected under interference.

Stayman Overlays (Concrete Example)

stayman-doubled

Activates when: CompetitionMode.Doubled + SystemMode.Modified

Action: Replaces hand tree with round1AskAfterDouble

Tree: 10+ HCP → PenaltyRedouble (redouble). 8-9 HCP + major → EscapeRescue (2C Stayman despite double). Below 8 HCP → CompetitivePass. This models standard Stayman after double: strong hands penalize, moderate hands escape, weak hands pass.

stayman-overcalled

Activates when: Non-uncontested + SystemMode.Off

Action: Replaces hand tree with fallback("system-off-overcall")

Effect: Convention produces no bid. The hand will be evaluated by the catch-all convention (SAYC) instead. This models the real-world behavior where Stayman is "off" after an overcall.

Validation at registration: validateOverlayPatches(overlays, protocol) checks that each overlay's roundName matches an actual round in the protocol. Also calls validateTree() on each replacementTree (Phase 6). Invalid round names or interleaved trees throw at registerConvention() time, not at runtime. Overlays without a protocol also throw.
Error handling: Every hook call in generateCandidates() is wrapped in try/catch. If any hook throws, the error is logged via onOverlayError callback (or console.warn) and the pipeline continues without that hook's effect. This prevents a buggy overlay from crashing the entire bidding system.

Matching via nodeId

The critical "which proposal is the matched one?" check uses nodeId comparison:

// From candidate-generator.ts
const isMatched = proposal.sourceNode !== undefined
  && proposal.sourceNode.nodeId === matched?.nodeId;  // nodeId identity!
Why nodeId, not reference identity? Multiple IntentNodes can share the same name (e.g., "stayman-rebid-minor-gf" appears for both diamonds and clubs). Name-based matching would be ambiguous. nodeId is auto-assigned by intentBid() via a monotonic counter, guaranteeing uniqueness within a process. It survives spread copies (unlike === reference identity). Overlay-injected proposals have no sourceNode, so they're never matched — this is intentional.

10. Dual Traversal Paths

The tree is traversed by two independent systems for different purposes. This is a deliberate architectural choice — the production decision path is decoupled from the display/teaching path.

Production / Strategy Path

What the AI actually bids

▶ collectIntentProposals()
▶ generateCandidates()
▶ selectMatchedCandidate()
Characteristics:
  • Overlay-aware (all 5 hooks applied)
  • Resolves through intent system
  • Checks call legality
  • Uses nodeId matching
  • Multi-encoding resolver support
  • Tiered selection (4 tiers)
  • Used by conventionToStrategy()

Display / Teaching Path

What the UI shows the user

▶ findSiblingBids()
▶ flattenTreeForDisplay()
▶ extractTeachingContent()
Characteristics:
  • Uses defaultCall only (unaware of intent resolvers)
  • Shows all alternatives with failed conditions
  • Enriches with teaching metadata
  • Optional overlay context (TreeDisplayOverlayContext)
  • Used by BidFeedbackPanel, RulesPanel, LearningScreen
Why two paths? The display path shows "what alternatives existed and why they didn't apply" — this is teaching content. The production path picks "what to actually bid" with overlay awareness and intent resolution. Coupling them would mean overlay hooks could break the teaching display, and teaching enrichment would add overhead to the hot decision path.

13. Edge Cases

Auction conditions after hand conditions

validateTree() walks every path at registration time. If it finds an auction condition below a hand condition, it throws immediately. This prevents interleaving bugs that would confuse both the inference engine (which relies on the auction/hand split) and the sibling finder (which assumes a clean hand subtree root).

// This would throw at registerConvention() time:
handDecision("hcp-check", hcpMin(8),
  decision("partner-bid", isOpener(), // BOOM: auction after hand
    intentBid(...), intentBid(...)
  ),
  fallback()
)

Dynamic handTree function throws

If a round's handTree function throws (e.g., unexpected established context), the protocol evaluator catches it, logs a warning, and returns no match. The convention gracefully yields to the next strategy in the chain.

Multiple IntentNodes with same name

Names like stayman-rebid-minor-gf appear for both diamonds and clubs in the same tree. This is safe: matching uses nodeId (auto-assigned, unique per intentBid() call), the sibling finder tracks each as a separate alternative with its own path conditions, and flattenTree produces separate rules for each path.

seatFilter fails but trigger matches

When seatFilter fails, the cursor still advances (the milestone happened) but activeRound is NOT updated. This means subsequent rounds can still match. Example: Bergen Round 1 (opening) trigger matches for everyone; seatFilter isOpener() passes for North but not South. South's activeRound stays null until Round 2 where seatFilter is isResponder().

Overlay-injected intents can't be "matched"

Proposals from addIntents() have no sourceNode, so isMatched is always false for them. They can be selected via the "preferred" or "alternative" tiers, but never via Tier 1 (matched). This is intentional — the matched node must come from the actual tree evaluation, not from an overlay hook.

2NT vs 1NT Stayman

Stayman works at both levels. The openingLevel in EstablishedContext (set by the Round 1 trigger) drives handTree function dispatch to different subtrees. Resolvers use staymanLevel(state) which derives the response level from state.familyId ("1nt" → level 2, "2nt" → level 3). Same node names, same intents, different concrete calls.

Bergen biddingRound(n) disambiguation

Bergen's opener rebid (Round 3) and second rebid (Round 4) have the same trigger conditions. The seatFilter uses biddingRound(n) to disambiguate: Round 3 requires biddingRound(1) (opener's first rebid), Round 4 requires biddingRound(2) (opener's second). Without this, both rounds would activate on the same milestone.

matchedIntentSuppressed flow

When a suppressIntent hook removes the matched IntentNode specifically, matchedIntentSuppressed is set to true. If no other candidate is selected (all tiers empty), conventionToStrategy() returns null rather than falling back to defaultCall. This signals "convention explicitly declines" rather than "convention doesn't apply," causing the strategy chain to try the next strategy (natural fallback or pass).

Resolver "declined" vs no resolver

A resolver returning { status: "declined" } excludes the candidate entirely — the intent is inexpressible. A Map miss (no resolver registered for that intent type) returns null from resolveIntent(), falling back to defaultCall. This distinction ensures SAYC (empty resolvers) always uses defaultCall, while a convention that registers a resolver can actively reject intents in certain contexts.

Two-pass backfill field precedence

When convention and baseline rules both want to set the same field, the convention rule always wins. applyBackfillEffect() checks alreadySet (populated from the convention pass) and only applies baseline values for fields the convention didn't touch. For mergeConventionData, convention keys take precedence over baseline keys via shallow merge.

14. Design Decisions

Binary hand decisions, not multi-way

Hand decisions are strictly binary (yes/no). The NO branch encodes structural negation — "the hand doesn't have 4+ hearts." Multi-way branching would lose this structural negation because there's no concept of "which branch was NOT taken." The "multi-way" display for user-facing alternatives is handled by the sibling finder walking all branches, not by the tree structure itself.

Trade-off: deeper trees for multi-option checks (e.g., checking all 4 suits requires 4 nested decisions) but clean negative inference data.

Auction before hand, always

The protocol layer handles all auction-context dispatch. By the time the hand tree is reached, every remaining condition is a hand condition. This strict ordering means the sibling finder can assume a "hand subtree root" exists, and the inference engine can cleanly separate auction path conditions from hand constraints. validateTree() enforces this at registration time.

Trade-off: can't express "if they doubled AND I have 10+ HCP" as a single condition. Instead, overlay hooks handle the "if doubled" part and the tree handles "10+ HCP".

Intent + defaultCall dual representation

Every IntentNode carries both a semantic intent AND a concrete defaultCall. This duality serves different consumers: the display/inference path uses defaultCall (fast, no resolver overhead); the production path uses intent + resolver (context-sensitive call generation). The defaultCall acts as the "obvious" concrete bid that works in the uncontested, normal case.

Trade-off: two representations can drift. Resolvers must return calls consistent with defaultCall's semantics, though the level may differ.

Two independent traversal paths

Production (intent collector + candidate generator) and display (sibling finder + flatten) traverse the same tree independently. This is intentional — the display path shows all alternatives for teaching while the production path applies overlays and resolvers for actual bidding. Coupling them would mean overlay hooks could break teaching content.

Downside: ~15 lines of logic duplicated (both skip auction conditions, walk hand subtree). The independence is worth it.

Registration-time validation

Four validators run at registerConvention() time: validateProtocol() (no empty rounds, all triggers are auction conditions), validateTree() (auction before hand on all paths, including overlay replacement trees), validateOverlayPatches() (round names exist in protocol), and two-pass rule ID disjointness check. This "fail fast" approach catches authoring errors at app startup, not during gameplay.

Trade-off: startup cost. In practice, validation runs in <1ms for all 4 conventions.

Overlay patching, not tree mutation

Interference is handled by overlays that patch behavior, not by adding interference branches into every convention tree. This keeps trees clean and focused on normal (uncontested) bidding, while overlays express "when doubled, do this instead." Overlays compose via priority ordering and well-defined hook semantics (first-wins for tree replacement, all-compose for suppression).

Trade-off: overlay logic is separate from the tree, making it harder to see the "complete" behavior at a glance. mitigated by TreeDisplayOverlayContext for teaching display.

Replay-based dialogue state

DialogueState is rebuilt from scratch by replaying the entire auction through transition rules. There's no incremental update. This eliminates state synchronization bugs and makes the system deterministic and testable. The two-pass model (convention rules first, baseline backfill second) ensures convention-specific semantics always take priority.

Trade-off: O(n) cost per evaluation. Acceptable because n < 20 for bridge auctions.

Strategy chain for graceful degradation

createStrategyChain() wraps convention strategies in a fallback chain: convention → natural fallback → pass. When a convention explicitly declines (overlay suppression, no match), the chain tries progressively simpler strategies. The natural fallback provides competitive protection (bid with 6+ HCP and a suit) without convention-specific logic.

Trade-off: additional strategy abstraction layer. Justified because it makes convention decline semantically clean (return null) rather than requiring every convention to handle its own fallback.

nodeId over object reference identity

Candidate matching uses nodeId (auto-assigned "intent-N") rather than JavaScript's === reference identity. This survives spread copies, tree reconstructions, and any other operation that creates new objects with the same data. The monotonic counter guarantees uniqueness within a process.

Trade-off: nodeId is process-scoped, not globally unique. Sufficient because all convention trees are built once at registration time and shared across evaluations.
