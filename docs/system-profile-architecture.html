System Profile & Base System Injection

How BaseSystemId propagates through the convention pipeline to produce system-aware module behavior at runtime.

TL;DR

The core idea: Convention modules are system-agnostic. They never import concrete HCP thresholds. Instead, the base system choice (BaseSystemId) flows through a parameterization chain:

BaseSystemId → SystemConfig → SystemProfile → closures in fact evaluators

Modules declare what they need (a stable fact ID like system.responder.inviteValues). The system provides the concrete answer at runtime via closure-captured SystemConfig. Same module code, different system behavior.

Mental Model

Think: dependency injection via closures.

In typical DI, a service declares an interface it needs and the container provides the concrete implementation. Here:

  • Interface = a fact ID string (e.g. SYSTEM_RESPONDER_INVITE_VALUES)
  • Container = createSystemFactCatalog(sys) which closes over SystemConfig
  • Consumer = a surface clause that references the fact ID in its conditions

Module attachments work similarly to CSS media queries — declarative conditions (auction pattern, capabilities) that toggle a module on/off without imperative logic.

1. Vocabulary: BaseSystemId

File: src/core/contracts/base-system-vocabulary.ts

Three string constants define the universe of supported bidding systems. Every reference to a system name in the codebase traces back here.

export const BASE_SYSTEM_SAYC = "sayc" as const;
export const BASE_SYSTEM_TWO_OVER_ONE = "two-over-one" as const;
export const BASE_SYSTEM_ACOL = "acol" as const;

export type BaseSystemId =
  | typeof BASE_SYSTEM_SAYC
  | typeof BASE_SYSTEM_TWO_OVER_ONE
  | typeof BASE_SYSTEM_ACOL;

Adding a new system starts here: add a constant, extend the union.

2. Parameterization: SystemConfig

File: src/core/contracts/system-config.ts

SystemConfig captures every threshold that varies between systems but is shared across all conventions within a system. Convention modules must not hardcode these values.

The lookup function resolves a BaseSystemId to its concrete config. Falls back to SAYC for unknown IDs.

const SYSTEM_CONFIG_MAP: Record<string, SystemConfig> = {
  [BASE_SYSTEM_SAYC]: SAYC_SYSTEM_CONFIG,
  [BASE_SYSTEM_TWO_OVER_ONE]: TWO_OVER_ONE_SYSTEM_CONFIG,
  [BASE_SYSTEM_ACOL]: ACOL_SYSTEM_CONFIG,
};

export function getSystemConfig(id: BaseSystemId): SystemConfig {
  return SYSTEM_CONFIG_MAP[id] ?? SAYC_SYSTEM_CONFIG;
}

3. Profile Creation: SystemProfile

Type: src/core/contracts/agreement-module.ts  |  Factory: src/conventions/core/profile-builder.ts

A SystemProfile is the activation contract for a convention bundle. It declares which modules are active and under what conditions.

classDiagram
    class SystemProfile {
        +string profileId
        +BaseSystemId baseSystem
        +SystemConfig systemConfig
        +ModuleEntry[] modules
    }
    class ModuleEntry {
        +string moduleId
        +ModuleKind kind
        +Attachment[] attachments
    }
    class Attachment {
        +AuctionPattern whenAuction
        +PublicGuard whenPublic
        +string[] requiresCapabilities
        +string[] requiresVisibleMeanings
    }
    SystemProfile "1" --> "*" ModuleEntry : modules
    ModuleEntry "1" --> "*" Attachment : attachments

The factory: createSystemProfile() auto-resolves SystemConfig from the baseSystem ID.

export function createSystemProfile(config: SystemProfileConfig): SystemProfile {
  return {
    profileId: config.profileId,
    baseSystem: config.baseSystem,
    systemConfig: config.systemConfig ?? getSystemConfig(config.baseSystem),  // auto-resolve
    modules: config.modules,
  };
}

4. Runtime Activation: resolveActiveModules()

File: src/conventions/core/runtime/profile-activation.ts

Each auction tick, the runtime asks: "given this profile and the current auction state, which modules are active?" Attachments are the declarative answer.

    style INPUT fill:#1e293b,stroke:#38bdf8,color:#e2e8f0

Attachment matching rules

  • whenAuction — sequence: auction starts with these calls. contains: auction includes a call (optionally by role). by-role: last call by a role matches.
  • requiresCapabilities — all named capabilities must be present in the active set.
  • whenPublic — a guard evaluated against the public snapshot's registers (exists, eq, neq, in).
  • requiresVisibleMeanings — all required meaning family IDs must have been committed to the public record.

Within a single attachment, all conditions are AND-ed. Across attachments on the same module, they are OR-ed (any one match activates the module).

5. Fact Injection: The Closure Pattern

Vocabulary: src/core/contracts/system-fact-vocabulary.ts  |  Catalog: src/conventions/pipeline/system-fact-catalog.ts

This is the key mechanism that makes modules system-agnostic. Modules reference stable fact IDs. The system provides concrete evaluators that close over SystemConfig.

sequenceDiagram
    participant Author as Convention Author
    participant Surface as Surface Clause
    participant Catalog as system-fact-catalog.ts
    participant Config as SystemConfig

    Author->>Surface: references fact ID "system.responder.inviteValues"
    Note over Author,Surface: Module never imports concrete thresholds

    Config->>Catalog: createSystemFactCatalog(sys)
    Note over Catalog: Closure captures sys.responderThresholds

    Surface->>Catalog: evaluate fact ID
    Catalog->>Catalog: hcp >= sys.responderThresholds.inviteMin
AND hcp <= sys.responderThresholds.inviteMax
    Catalog-->>Surface: true/false
    Note over Surface: Same module, different thresholds per system

Two flavors of system facts

Hand-dependent

Combine hand data (HCP) with system thresholds. Value changes per hand AND per system.

// "Is this 11-HCP hand invite-strength?"
// SAYC: yes (8-9 invite) -> no, 11 > inviteMax
// Acol: yes (10-12 invite) -> yes
[SYSTEM_RESPONDER_INVITE_VALUES,
 (_h, _ev, m) => {
  const hcp = num(m, "hand.hcp");
  return fv(SYSTEM_RESPONDER_INVITE_VALUES,
    hcp >= sys.responderThresholds.inviteMin
    && hcp <= sys.responderThresholds.inviteMax);
}]

System-constant

Pure system properties, same for every hand. Let surfaces gate on system rules.

// "Is a 2-level new suit game-forcing?"
// SAYC: no (one-round)
// 2/1: yes (game)
[SYSTEM_SUIT_RESPONSE_IS_GAME_FORCING,
 () => fv(
  SYSTEM_SUIT_RESPONSE_IS_GAME_FORCING,
  sys.suitResponse.twoLevelForcingDuration
    === "game"
)]

Why closures, not interface methods?

The evaluator signature (FactEvaluatorFn) doesn't include SystemConfig as a parameter. Instead, createSystemEvaluators(sys) captures sys in the closure scope. This keeps the evaluator API uniform across all fact layers (bridge-level, system-level, convention-level) while letting system facts access their config.
