Module Architecture & Coupling Analysis

Pre-refactor analysis of 11 modules, ~10,200 LOC across ~103 production files. No circular dependencies. Five coupling hotspots identified for decoupling.

Tauri 2 + Svelte 5 + TypeScript strict Generated 2026-03-07

Executive Summary

The src/ directory contains 11 modules organized as a clean directed acyclic graph (DAG). No runtime circular dependencies exist. The architecture follows a layered pattern: a pure engine foundation, domain logic in the middle tier, and UI at the top.

0
Circular dependencies
4 layers

5
Coupling hotspots to address

Bottom line for refactoring: The module boundaries are sound. The main decoupling opportunities are (1) moving bid-suggester.ts out of the engine, (2) creating a conventions facade so display/ stops reaching into convention internals, (3) breaking the shared/types.ts dependency on BiddingContext, (4) relocating a misplaced integration test, and (5) reducing drill/'s fan-out from 5 modules. None of these are urgent blockers, but each would make future extraction into separate packages cleaner.

Coupling Hotspots

Five areas where module boundaries are weaker than the rest of the codebase. Ranked by refactoring impact.

#1

display/ reaches into conventions/core/ internals

display/rules-display.ts imports from 3 separate files inside conventions/core/: condition-evaluator, context-factory, and registry. display/debug-bid-eval.ts adds debug-utils and registry. Total: 5 internal imports bypassing any facade.

Encapsulation break Files: src/display/rules-display.ts:3-14, src/display/debug-bid-eval.ts:1-6

Impact: Any internal refactor of conventions/core/ breaks display/. If conventions were extracted as a separate package, display/ would need access to internals not in the public API.

#2

engine/bid-suggester.ts has type-only but semantic coupling to conventions

bid-suggester.ts lives in engine/ (the "zero cross-module imports" foundation) but uses import type { BiddingContext } from conventions/core/types and manually reconstructs a BiddingContext object inline (lines 21-28), duplicating the shape from context-factory.ts. The file contains the warning: "Update defaults here if BiddingContext changes."

Semantic coupling File: src/engine/bid-suggester.ts:1-28

Impact: If BiddingContext adds required fields, the inline construction silently produces an incomplete object. This file functionally belongs in strategy/, not engine/.

#3

shared/types.ts depends on BiddingContext from conventions

shared/types.ts line 11: import type { BiddingContext } from "../conventions/core/types". The BiddingStrategy.suggest() method signature takes BiddingContext as a parameter, creating a type-level dependency from the shared DTO layer upward to conventions.

Upward type dependency File: src/shared/types.ts:11

Impact: Any change to BiddingContext ripples through shared → strategy → drill → stores. This is the hardest seam to break if conventions were ever extracted to a separate package.

#4

drill/ has 5-module fan-out for a 327 LOC module

drill/ imports from engine, conventions, inference, strategy, and shared. With only 4 files and 327 LOC, it has the highest dependency-to-size ratio in the codebase. It acts as a wiring/composition root for the drill lifecycle.

High fan-out Files: src/drill/config-factory.ts, src/drill/helpers.ts

Impact: Acceptable for a composition root, but config-factory.ts hard-codes strategy and inference provider selection. A configuration object or factory injection could reduce this coupling.

#5

conventions/__tests__/sayc/drillability.test.ts imports from display/ and drill/

A test file inside conventions/__tests__/ imports filterConventions from display/ and createDrillConfig from drill/. This creates a test-time upward dependency from a lower-layer module's test suite to higher-layer modules.

Misplaced test File: src/conventions/__tests__/sayc/drillability.test.ts:9-10

Impact: If drill/ or display/ breaks, convention tests fail — violating layer isolation. This test should live in drill/__tests__/ or a top-level integration test directory.

Methodology

Import counting: All cross-module from statements in production .ts and .svelte files, excluding __tests__/, .test. files, and test-support/. Each import line counted as 1, regardless of how many symbols it imports.

Type-only detection: Imports using import type { ... } syntax. These create compile-time dependencies only — they are erased at build time and do not create runtime coupling.

Circular dependency check: Full topological analysis of the module graph. Confirmed DAG with no cycles at either runtime or type level.

Tools: grep for import extraction, manual verification of key files, CLAUDE.md context files for each module.
