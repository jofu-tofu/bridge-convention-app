How the Backend Pipeline Works

A plain-English guide to the four layers that turn convention rules into bid recommendations.

The Big Picture

The backend's job is one thing: given a player's hand and the auction so far, recommend the best bid according to the active convention.

It does this through four layers, each with a distinct responsibility:

Layer 1 — Config & Bundle

Defines what conventions exist and packages them into usable units.

Layer 2 — Bid Meanings

Each possible bid meaning is described as a set of hand requirements.

Layer 3 — The Pipeline

Evaluates every candidate meaning against the actual hand and picks a winner.

Layer 4 — Convention Protocol

The formal architecture for state machines, registers, and layered modules.

flowchart LR
  A["Convention Modules\n(LEGO bricks)"] --> B["Bundle\n(kit box)"]
  B --> C["Active Bid Meanings\n(candidates)"]
  C --> D["Pipeline\n(screening + judging)"]
  D --> E["Recommended Bid"]

Analogy: Think of it like building with LEGO. Individual convention modules are bricks. A bundle snaps them into a kit. The pipeline is the instruction manual that decides which brick to place next, given the current state of the build.

Layer 1: Convention Config & Bundle

Problem this solves: How do you define a convention like Stayman, package it with related conventions (Jacoby Transfers, Smolen), and make it selectable in the UI?

Convention Modules: The LEGO Bricks

A module (ConventionModule) is a self-contained unit of bidding logic. Each module owns:

  • Bid meanings — the possible bids it contributes (covered in Layer 2)
  • Facts — hand properties it cares about (e.g., "has a 4-card major")
  • FSM states — where the conversation is in this module's flow
  • Teaching metadata — explanations for the UI

Hard rule: modules never import from each other. Stayman doesn't know Jacoby Transfers exist. All cross-module wiring happens one level up, in the bundle.

Bundles: The Kit Box

A bundle (ConventionBundle) groups modules into a practicable bidding system. It's the unit the UI presents as "1NT Responses" or "Bergen Raises."

A bundle adds things a single module can't provide alone:

  • System config — shared parameters like "1NT = 15–17 HCP"
  • Deal constraints — rules for generating practice hands
  • Surface router — decides which surfaces are active given the auction
  • Teaching content — cross-module relationships ("Stayman is an alternative to Transfers")

Convention Config: Auto-Derived from the Bundle

ConventionConfig is the lightweight, UI-facing metadata extracted from a bundle. It's what the convention picker screen displays: name, description, category.

The config is auto-derived — there's no separate config file. When you call registerBundle(), it automatically extracts a ConventionConfig via createConventionConfigFromBundle() and registers both.

The flow is: Modules → buildBundle() → Bundle → registerBundle() → auto-derives Config.

composeBundles() — merging multiple kits

When the app needs to combine separate bundles (e.g., NT + Bergen), composeBundles() merges them. Surfaces are concatenated. Deal constraints take the most restrictive values. Explanation catalogs are deduplicated. If both bundles have a conversation machine, the first one wins.

Layer 2: Bid Meanings

Problem this solves: How do you formally describe what a bid means so the system can evaluate whether it applies to a given hand?

Analogy: A BidMeaning is like a job posting. It says: "I am the bid 2C, and I'm the right choice IF you have 8+ HCP AND a 4-card major AND no 5-card major." The hand is the applicant; the clauses are the job requirements.

Anatomy of a BidMeaning

Every surface has five key parts:

1. Identity — meaningId (unique name like "stayman:ask-major") and semanticClassId (equivalence group for deduplication).

2. Encoding — the actual bid. encoding.defaultCall is the call (e.g., 2C). Some surfaces have alternate encodings for relay systems.

3. Clauses — the hand requirements. Each clause checks one fact (HCP, suit length, shape) against a threshold using an operator (gte, lte, range, boolean).

4. Ranking — how important this bid is. The recommendationBand is "must", "should", "may", or "avoid". Specificity (how narrow the requirements are) is calculated automatically by the pipeline.

5. Teaching — teachingLabel (human name like "Stayman 2C"), sourceIntent (bridge semantics like "StaymanAsk"), and teachingTags (links to related bid meanings).

How Bid Meanings Are Created

Convention authors use the createSurface() builder. It takes simplified inputs and returns a complete BidMeaning with derived fields filled in automatically:

  • clauseId — auto-derived from factId + operator + value
  • description — auto-generated human text (e.g., "HCP at least 12")
  • modulePrecedence — stamped by the bundle's composition layer, not the author
  • specificity — derived by the pipeline from clause dimensions, never hand-authored

Clauses: The Requirements Checklist

Each clause references a fact — a measurable property of the hand. Facts are organized in a catalog:

Fact CategoryExample FactsWhat They Measure
Primitivehand.hcp, hand.suitLength.heartsDirect hand analysis (point count, suit lengths)
Bridge-derivedbridge.hasFourCardMajor, bridge.isBalancedDerived from primitives (has 4+ hearts or spades?)
Module-derivedstayman.hasStaymanPointsConvention-specific calculations
Relationalbridge.supportCount.$suitDepends on auction context (support for partner's suit)

Clause operators work like you'd expect:

  • gte — greater than or equal ("at least 12 HCP")
  • lte — less than or equal ("at most 17 HCP")
  • range — between two values ("12–14 HCP")
  • boolean — true/false ("is balanced? true")
  • in — one of a set ("suit is hearts or spades")

Layer 3: The Meaning Pipeline & Arbitration

Problem this solves: Given a hand, many surfaces could apply. The pipeline screens them, evaluates them, and picks the single best bid.

Analogy: The pipeline is a hiring process. Multiple candidates (surfaces) apply for the "recommended bid" position. They go through a screening funnel — are they qualified? Is the position open? — then a ranking panel picks the best one.

The Five Pipeline Stages

flowchart LR
  S1["1. Select\nSurfaces"] --> S2["2. Compose\n(Transforms)"]
  S2 --> S3["3. Evaluate\nFacts"]
  S3 --> S4["4. Evaluate\nMeanings"]
  S4 --> S5["5. Arbitrate\n(Pick Winner)"]

Stage 1: Select Surfaces

What happens: The rule interpreter (collectMatchingClaims()) looks at the auction history and asks each module: "Given where we are in the conversation, which of your surfaces are candidates right now?"

How it decides: Each module has a local state machine (phases like "idle" → "asked" → "shown-hearts"). The interpreter replays the auction through each module's phases, then collects bid meanings whose rules match the current turn (opener/responder/opponent), phase, and negotiation state (fit agreed? forcing? who's captain?).

Code NamePlain English
collectMatchingClaims()Ask each module: "What bids do you offer right now?"
deriveTurnRole()Figure out if the next bidder is opener, responder, or an opponent
PhaseWhere a module is in its local conversation flow (e.g., "asked", "denied")
NegotiationStateThe shared negotiation state: has a fit been agreed? Is the auction forcing?

Stage 2: Compose Surfaces (Transforms)

What happens: composeSurfaces() applies structural transforms to the candidate pool before evaluation. Three kinds:

  • Suppress — Remove a surface from consideration ("When Smolen is active, suppress the natural 3H bid")
  • Inject — Add a new surface that wasn't in any module ("Insert an emergency 3NT surface")
  • Remap — Change a surface's encoding ("In this context, the bid that normally uses 2D should use 3D instead")

Stage 3: Evaluate Facts

What happens: evaluateFacts() analyzes the player's actual hand and produces a dictionary of fact values.

Facts are evaluated in dependency order (topologically sorted). Three tiers run sequentially:

  1. Standard facts — HCP, suit lengths, honor counts
  2. Relational facts — facts that depend on auction context (e.g., "support count for partner's declared suit")
  3. Posterior facts — facts inferred from partner's previous bids

Output: a map of factId → value (e.g., "hand.hcp" → 14).

Stage 4: Evaluate Meanings

What happens: evaluateAllBidMeanings() takes each bid meaning and checks every clause against the actual facts. A bid meaning becomes a MeaningProposal — the same structure, but now each clause is marked satisfied: true/false with the observedValue.

This stage also derives specificity — how narrow the surface's requirements are. More dimensions constrained (suit identity + point range + shape) = higher specificity. This is never hand-authored; it's computed from the clause structure.

Code NamePlain English
MeaningProposalA bid meaning after evaluation — each clause now has a pass/fail verdict
satisfiedDid this hand requirement pass?
observedValueWhat the hand's actual value was (e.g., HCP = 14)
specificityHow narrow the requirements are (more constraints = more specific)
ConstraintDimensionWhat type of info a clause communicates: suit identity, point range, shape, etc.

Stage 5: Arbitration (Pick the Winner)

What happens: arbitrateMeanings() takes all proposals and selects the single best bid through a 5-step process:

flowchart TD
  G["All Proposals"] --> G1["Gate 1: Semantic\nAll clauses satisfied?"]
  G1 -->|pass| G2["Gate 2: Encoding\nCan this be bid?"]
  G2 -->|pass| G3["Gate 3: Legality\nIs the call legal here?"]
  G3 -->|pass| TS["Truth Set\n(qualified candidates)"]
  G1 -->|fail| EL["Eliminated\n(with reason)"]
  G2 -->|fail| EL
  G3 -->|fail| EL
  TS --> RK["Rank by:\n1. Band\n2. Specificity\n3. Module Precedence\n4. Intra-Module Order"]
  RK --> DD["Deduplicate by\nSemantic Class"]
  DD --> W["Winner\n(first in ranked list)"]

The Four Gates

Every proposal must pass through these gates in order. Fail one, and you're eliminated:

GateCode NameQuestion It Asks
1semantic-applicabilityAre all clauses satisfied by this hand?
2obligation-satisfaction(Reserved for future use — always passes)
3encoder-availabilityCan the system encode this meaning as a concrete bid?
4concrete-legalityIs this call legal in the current auction state?

The Ranking Order

Among proposals that pass all gates (the truth set), the winner is chosen by this tiebreaker chain:

  1. recommendationBand — must beats should beats may beats avoid. A "must" bid always wins over a "should" bid, regardless of other factors.
  2. specificity — higher wins. A surface requiring 5 constraints beats one requiring 2, because it's a more precise match.
  3. modulePrecedence — lower number wins. Controls cross-module priority (e.g., Stayman before natural bids).
  4. declarationOrder — lower number wins. Last-resort deterministic tiebreaker within a single module.

Semantic Class Deduplication

After ranking, if two surfaces resolve to the same semanticClassId (meaning they're fundamentally the same bid from different modules), only the higher-ranked one survives.

Code NamePlain English
truthSetAll proposals that passed every gate (hand fits + legal)
acceptableSetProposals that are legal but don't fully fit the hand
recommendedThe truth set after deduplication and ranking
selectedThe single winning proposal (first in recommended list)
eliminationsLog of why each rejected proposal was eliminated
provenanceComplete audit trail of every decision the pipeline made
evidenceBundleStructured evidence package for the teaching UI

Layer 4: Convention Protocol

Problem this solves: How do you model complex, multi-round, overlapping conventions with state that evolves through the auction?

Analogy: The protocol layer is a franchise operating manual. It defines the formal type contracts for how convention modules can be structured: what states they can be in, what triggers transitions, what data they can read and write, and how multiple active conventions coexist without conflicts.

Current status: The protocol types (ConventionSpec, ModuleSpec, etc.) define the formal architecture. The active runtime uses the rule interpreter + meaning pipeline (Layer 3) rather than the protocol's own replay engine. The protocol types remain as the canonical specification language.

Two Module Roles

The protocol architecture distinguishes two kinds of modules:

Base Module

The primary conversation flow (e.g., "we opened 1NT"). Exactly one base module is active per conversation. Selected by the opening bid pattern.

Code: role: "base"

Protocol Module

A convention layered on top of the base (e.g., Stayman activates on top of the NT base). Multiple protocols can be active simultaneously. Activated by guard conditions.

Code: role: "protocol"

Key Protocol Concepts

Code NamePlain English

The Runtime Snapshot

At any point in the auction, the system's state is a RuntimeSnapshot — a sparse vector containing:

  • Boot node — where we are in the opening-pattern trie
  • Active base track — which base module is selected (e.g., "1NT system")
  • Active protocols — which protocol modules are attached, in stack order
  • Registers — shared data with provenance (who wrote it, when)
  • Active tags — boolean flags exported by states and effects
  • Done latches — protocols that finished and can't re-attach

How Surfaces Compose Across Layers

Each active module contributes a SurfaceFragment. Fragments specify how they interact with layers below them:

RelationMeaningExample
"augment"Add rules; lower layers still visibleStayman adds 2C option alongside natural bids
"compete"Both layers visible; ranking decides which winsTwo modules offer different meanings for 2D
"shadow"Covered actions from lower layers are hiddenTransfer protocol shadows the natural 2D meaning

End-to-End: From Hand to Recommended Bid

Here's what happens when you're dealt a hand with 11 HCP, 4 hearts, 3 spades, 3 diamonds, 3 clubs, and partner opened 1NT:

sequenceDiagram
    participant U as Your Hand
    participant RI as Rule Interpreter
    participant FC as Fact Evaluator
    participant ME as Meaning Evaluator
    participant AR as Arbitrator
    participant UI as UI

    U->>RI: Hand + Auction (1NT - Pass)
    RI->>RI: Replay auction through each module's phases
    RI->>RI: Stayman: phase=idle, turn=responder -> match
    RI->>RI: Transfers: phase=idle, turn=responder -> match
    RI->>RI: Natural NT: phase=idle, turn=responder -> match
    RI-->>FC: 12 candidate surfaces

    FC->>FC: Evaluate hand facts
    Note over FC: hand.hcp = 11, hand.suitLength.hearts = 4,
bridge.hasFourCardMajor = true,
bridge.hasFiveCardMajor = false

    FC-->>ME: Facts dictionary
    ME->>ME: Check each surface's clauses against facts
    Note over ME: Stayman 2C: all clauses pass
Transfer 2D: FAIL (no 5-card heart suit)
2NT invite: all clauses pass
3NT game: FAIL (need 10+ HCP? has 11, but needs 15 total)

    ME-->>AR: Proposals with pass/fail verdicts
    AR->>AR: Gate check (semantic + legality)
    AR->>AR: Truth set: Stayman 2C, 2NT invite
    AR->>AR: Rank: Stayman (should, specificity=3) beats 2NT (may, specificity=2)
    AR-->>UI: Recommend Stayman 2C

Why Stayman wins over 2NT:

  • recommendationBand: Stayman is "should"; 2NT is "may". "Should" ranks higher.
  • Even if both were "should", Stayman has higher specificity (constrains suit identity + point range + shape vs. just point range).
  • Transfer to hearts was eliminated at the semantic gate: the hand only has 4 hearts, but the surface requires 5.
