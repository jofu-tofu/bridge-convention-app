Architecture Overview

7 backend modules form an acyclic dependency graph, enforced at three levels: TypeScript interfaces, barrel exports, and ESLint rules. The service/ module is the sole hexagonal port — the only gateway the UI and CLI may use to reach backend logic.

Think of it like an embassy: all communication between citizens (UI components) and the foreign government (game engine, conventions, inference) passes through official diplomatic channels (ServicePort). Citizens never contact foreign departments directly.

Key invariant: No circular dependencies exist. The graph is a strict DAG. Engine is a leaf — it imports nothing from other backend modules.

Dependency Graph

Each module has explicit allowed and blocked imports, enforced by ESLint no-restricted-imports rules in eslint.config.js.

Module Path Imports From Blocked
engine src/engine/ none (leaf) svelte, stores, components, strategy, conventions, inference, bootstrap
conventions src/conventions/ engine svelte, stores, components, bootstrap
inference src/inference/ engine, conventions svelte, stores, components, bootstrap
strategy src/strategy/ engine, conventions, inference svelte, stores, components, bootstrap
bootstrap src/bootstrap/ engine, conventions, strategy, inference svelte, stores, components
service src/service/ engine, conventions, strategy, inference, bootstrap svelte, stores, components

The dependency direction is strictly downward: cli → service → bootstrap → strategy → inference → conventions → engine. No upward arrows exist.

Interface Contracts

Five key interfaces define the contracts between modules. Each sits at a boundary where one module's implementation meets another's expectations.

ServicePort — src/service/port.ts

The hexagonal boundary between UI/CLI and all backend logic. Every user action enters here.

interface ServicePort {
  createSession(config: SessionConfig): Promise<SessionHandle>
  destroySession(handle: SessionHandle): Promise<void>
  startDrill(handle: SessionHandle): Promise<DrillStartResult>
  submitBid(handle: SessionHandle, call: Call): Promise<BidSubmitResult>
  acceptPrompt(handle: SessionHandle): Promise<PromptAcceptResult>
  playCard(handle: SessionHandle, card: Card): Promise<PlayCardResult>
  getViewport(handle: SessionHandle): Promise<SessionViewport>
}

interface DevServicePort extends ServicePort {
  getExpectedBid(handle): Promise<{ call: Call } | null>
  getDebugSnapshot(handle): Promise<ServiceDebugSnapshot>
}

EnginePort — src/engine/port.ts

Async boundary allowing multiple transports (WASM for browser, Tauri IPC for desktop). All methods return Promise<T>.

interface EnginePort {
  generateDeal(constraints: DealConstraints): Promise<Deal>
  evaluateHand(hand: Hand): Promise<HandEvaluation>
  solveDDS(deal: Deal): Promise<DDSolution>
}

BiddingStrategy — src/strategy/bidding/bidding-types.ts

Strategy pattern. Multiple implementations (meaning-based, natural fallback, pragmatic) are chained via createStrategyChain().

interface BiddingStrategy {
  suggest(context: BiddingContext): Promise<BidResult | null>
}

interface BiddingContext {
  readonly hand: Hand
  readonly auction: Auction
  readonly seat: Seat
  readonly evaluation: HandEvaluation
  readonly vulnerability?: Vulnerability
}

ConventionModule — src/conventions/core/convention-module.ts

The unit of convention authoring. Modules are like LEGO bricks — portable, self-contained, composable into any bundle. They receive SystemConfig via factory parameter, never import concrete configs.

interface ConventionModule {
  readonly moduleId: string
  readonly facts: FactCatalogExtension
  readonly explanationEntries: Record<string, ExplanationEntry>
  readonly local: LocalFsm
  readonly states: StateEntry[]
}

// Factory pattern: system-agnostic creation
const moduleFactory = (sys: SystemConfig): ConventionModule => ({ ... })

InferenceEngine — src/inference/inference-engine.ts

Incremental per-bid processing with per-partnership information asymmetry. An InferenceCoordinator coordinates NS and EW engines.

interface InferenceEngine {
  processBid(entry: AuctionEntry, hand?: Hand): InferenceSnapshot
  getPublicBeliefState(): PublicBeliefState
}

interface InferenceSnapshot {
  readonly entry: AuctionEntry
  readonly newConstraints: readonly FactConstraint[]
  readonly cumulativeBeliefs: Record<Seat, PublicBeliefs>
}

Boundary Enforcement

Three enforcement layers prevent boundary violations.

1. TypeScript Interfaces

Port interfaces (ServicePort, EnginePort) define method contracts. Callers depend on the interface, not the implementation.

2. Barrel Exports

Each module exposes a single index.ts barrel. Deep imports into conventions/core/, service/evaluation/, etc. are forbidden.

3. ESLint Rules

no-restricted-imports rules in eslint.config.js block cross-boundary imports at lint time. ~250 lines of boundary rules.

Viewport Boundary

The service layer translates internal types into viewport-safe types that never leak implementation details to the UI. This is what the UI sees vs. what stays hidden:

Crosses the boundary

  • BiddingViewport, PlayingViewport
  • ViewportBidFeedback, TeachingDetail
  • Call, Card, Seat, Hand
  • GamePhase, SessionHandle

Never crosses

  • Deal (opponent hands hidden)
  • BidResult, ArbitrationResult
  • ConventionStrategy, StrategyEvaluation
  • DrillSession, InferenceEngine

Data Flow: A Bid

A single user bid touches 5 modules before the UI updates. Here is the path from button click to viewport refresh:

sequenceDiagram
    participant UI as UI (store)
    participant SVC as service
    participant BOOT as bootstrap
    participant STRAT as strategy
    participant CONV as conventions
    participant INF as inference
    participant ENG as engine

    UI->>SVC: submitBid(handle, Call)
    SVC->>ENG: evaluateHand(hand)
    ENG-->>SVC: HandEvaluation
    SVC->>STRAT: strategy.suggest(BiddingContext)
    STRAT->>CONV: runPipeline(facts, surfaces)
    CONV->>ENG: fact evaluation (HCP, shape)
    CONV-->>STRAT: PipelineResult
    STRAT-->>SVC: BidResult
    SVC->>BOOT: assembleBidFeedback()
    BOOT-->>SVC: BidFeedbackDTO
    SVC->>INF: coordinator.processBid()
    INF-->>SVC: InferenceSnapshot
    SVC->>SVC: buildViewport() + buildTeachingDetail()
    SVC-->>UI: BidSubmitResult (viewport-safe)
Phase transitions: After the auction completes (3 consecutive passes), the service transitions through BIDDING → DECLARER_PROMPT → PLAYING → EXPLANATION. Each phase builds its own viewport type.

Factory pattern: Each module exports a moduleFactory(sys: SystemConfig): ConventionModule. The same module works with SAYC, 2/1, or Acol without code changes.

ESLint enforcement: no-restricted-imports blocks sibling-module imports (e.g., bergen/ importing from ../dont/). This is in eslint.config.js lines 479-532.

Registration: Bundles register via registerConvention() in conventions/registration.ts. No hardcoded switch statements.

Type Ownership

After the dissolve-core refactoring, every type lives in its owning module. The former src/core/contracts/ directory was eliminated; types were distributed to the modules that define their semantics.

Types Owner Crosses Service Boundary?
Card, Hand, Deal, Suit, Rank, Seat, Call, Auction, Contract engine/types.ts Yes (universal vocabulary)
BidMeaning, MeaningClause, PipelineResult, ArbitrationResult conventions/pipeline/ No
FactCatalog, FactConstraint, ConventionModule, SystemProfile conventions/core/ No
TeachingProjection, BidGrade, ParseTreeView conventions/teaching/ Partially (BidGrade yes, projections no)
BiddingStrategy, BidResult, BiddingContext strategy/bidding/ No
InferenceSnapshot, PublicBeliefState, FactorGraph inference/ Partially (PublicBeliefState yes)
DrillConfig, DrillSession, DrillBundle, DrillTuning bootstrap/ No
BiddingViewport, PlayingViewport, ViewportBidFeedback, TeachingDetail service/ Yes (viewport-safe by design)
