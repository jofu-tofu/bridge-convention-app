Pipeline Overview

Given an auction context (auction history + hand + evaluation + seat + opponent conventions), the system produces a BidResult containing the recommended call, an explanation, a teaching trace, inference data, and an optional practical recommendation. The pipeline is fully deterministic: same inputs always produce the same output. Non-determinism exists only in deal generation (PRNG), never in evaluation.

The pipeline in one sentence:

Strategy Chain iterates strategies (convention adapter, natural fallback, pass) trying each in order. The convention adapter runs: Protocol Dispatch (sequential round matching) → Tree Evaluation (iterative binary traversal) → Dialogue State (auction replay) → Effective Context (bundle) → Intent Collection (recursive DFS) → Overlay Hooks (suppress/add/override) → Candidate Resolution (resolver calls) → Tiered Selection (matched > preferred > alternative) → Practical Scoring (weighted formula with belief data).

1. Entry Point: conventionToStrategy()

The entry point is conventionToStrategy(config, options?) in strategy/bidding/convention-strategy.ts. It wraps a ConventionConfig as a BiddingStrategy object with a single suggest(context): BidResult | null method. Returning null means "no opinion" -- the strategy declines to bid, and the chain tries the next strategy.

Algorithm: conventionToStrategy.suggest(context)

  1. Evaluate bidding rules: Call evaluateBiddingRules(context, config) -- this runs Protocol Dispatch + Tree Evaluation. If it returns null, the convention doesn't apply. Return null.
  2. Guard: If the tree matched an IntentNode, proceed to the candidate pipeline. Otherwise, use the flat rule result directly.
  3. Obtain belief data: Call options.beliefProvider(context) if provided (exceptions caught, yielding undefined).
  4. Build EffectiveConventionContext: Bundles raw context + config + protocol result + dialogue state + active overlays + belief.
  5. Generate candidates: generateCandidates(treeRoot, treeEvalResult, effectiveCtx) -- runs intent collection, overlay hooks, resolver calls.
  6. Compose ranker: config.rankCandidates first, then options.ranker. Both compose: config output feeds into options.
  7. Select candidate: selectMatchedCandidate(candidates, ranker, forcingState) -- tiered selection.
  8. Handle result: If selected, use its resolvedCall. If not selected and matchedIntentSuppressed, return null. Otherwise fall back to defaultCall.
  9. Compute practical recommendation: If belief data exists, score candidates + generate pragmatic candidates + build recommendation.
  10. Assemble BidResult: call, ruleName, explanation, meaning, handSummary, conditions, treePath, evaluationTrace, treeInferenceData, practicalRecommendation.

State produced: BidResult -- the complete output of evaluation for one bid decision.

2. Protocol Dispatch

Every convention defines a ConventionProtocol -- an ordered array of ProtocolRound objects representing the multi-round conversation structure of the convention (e.g., Stayman has 3 rounds: opening 1NT, response 2C, rebid). The protocol evaluator walks these rounds sequentially (linear scan of the rounds array), testing trigger conditions against the auction.

Algorithm: evaluateProtocol(proto, context, triggerOverrides?)

  1. Compute role: computeRole(entries, seat) scans the auction entries array linearly to determine if the evaluating seat is opener, responder, competitive, or rebidder.
  2. Initialize: cursor = 0, established = { role }, activeRound = null.
  3. For each round in proto.rounds (linear scan):
    1. Create a windowed context from cursor position: entries.slice(0, cursor + 2).
    2. Get effective triggers: overlay trigger overrides if provided, else the round's own triggers.
    3. Linear scan the triggers array. For each trigger, test trigger.condition.test(cursorCtx). First match wins.
    4. If no trigger matches, break -- auction hasn't reached this round.
    5. If matched: advance cursor += 2 (partnership turn = 2 seats), merge trigger.establishes into accumulated context.
    6. Seat filter check: If the round has a seatFilter, test it against the full context (not windowed). If it passes, set activeRound = r. If it fails, the milestone happened but this seat doesn't act -- cursor still advanced.
  4. Evaluate hand tree: If activeRound is set, resolve its handTree (may be static or a function of established context), then call evaluateTree(handTree, context).
stateDiagram-v2
    [*] --> ComputeRole
    ComputeRole --> InitCursor: role determined
    InitCursor --> TestRound: cursor=0
    TestRound --> WindowContext: for each round
    WindowContext --> TestTriggers: entries.slice(0, cursor+2)
    TestTriggers --> NoMatch: no trigger passes
    TestTriggers --> AdvanceCursor: first trigger passes
    NoMatch --> EvalHandTree: break loop
    AdvanceCursor --> SeatFilter: cursor += 2
    SeatFilter --> SetActive: seatFilter passes
    SeatFilter --> SkipActive: seatFilter fails
    SetActive --> TestRound: next round
    SkipActive --> TestRound: next round
    EvalHandTree --> [*]: return ProtocolEvalResult

Key insight: The cursor mechanism separates "what happened in the auction" from "who should act now." A round can match (advancing the cursor) without the current seat being the one who acts (seatFilter fails). This handles multi-round conventions where different partnership members act in different rounds.

Trigger condition types
  • bidMade(level, strain) -- seat-agnostic milestone (any player bid this)
  • partnerBidMade(level, strain) -- actor-aware: only fires for partner bids (uses areSamePartnership())
  • opponentBidMade(level, strain) -- actor-aware: only fires for opponent bids
  • doubleMade() -- seat-agnostic double detection
  • Combined with and(), or(), not() compositors

Seat filter patterns: isResponder(), isOpener(), not(seatHasActed()), biddingRound(n), passedAfter(level, strain)

3. Tree Evaluation: Binary Decision Traversal

The hand tree is a binary decision tree with three node types. Traversal is iterative (not recursive) via an infinite loop with a switch statement -- stack-safe for arbitrarily deep trees.

DecisionNode

Has condition (test + describe), yes branch, no branch. Can be auction or hand condition.

IntentNode

Leaf. Carries SemanticIntent, defaultCall, meaning, nodeId. Terminal match.

FallbackNode

Leaf. No match -- convention doesn't apply to this hand/auction. Optional reason.

Algorithm: evaluateTree(tree, context) -- Iterative Linear Path Traversal

node = tree
path = [], rejectedDecisions = [], visited = []

loop forever:
  switch node.type:
    case "intent":
      return { matched: node, path, rejectedDecisions, visited }

    case "fallback":
      return { matched: null, path, rejectedDecisions, visited }

    case "decision":
      passed = node.condition.test(context)
      description = node.condition.describe(context)
      entry = { node, passed, description }
      visited.push(entry)

      if passed:
        path.push(entry)        // record as passing
        node = node.yes         // follow YES branch
      else:
        rejectedDecisions.push(entry)  // record as rejected
        node = node.no                 // follow NO branch

Data structure: The tree is a linked structure (each node points to its children), not an array. Traversal follows exactly one path from root to leaf. This is neither DFS nor BFS -- it's a single-path walk through a binary decision tree. Every decision node is visited exactly once on the path. The tree never backtracks.

Output: TreeEvalResult contains four things:

  • matched -- the IntentNode reached (or null if FallbackNode)
  • path -- only the decision nodes where condition passed (YES branches taken)
  • rejectedDecisions -- only the decision nodes where condition failed (NO branches taken)
  • visited -- all decision nodes in traversal order (passing and rejected interleaved)

Validation invariant: validateTree() at registration time enforces that auction conditions always precede hand conditions on every path (DFS walk checking). This guarantees that findHandSubtreeRoot() can skip past auction conditions to find where hand evaluation begins.

4. Dialogue State: Auction Replay

Dialogue state is protocol memory -- what the auction means under the partnership agreement, independent of any hand. It's computed by replaying the entire auction one bid at a time through transition rules. Fully deterministic and stateless: same auction always produces the same DialogueState.

DialogueState contains:

FieldWhat it tracks
familyIdWhich convention family is active (e.g., "stayman")
forcingStateNonforcing | ForcingOneRound | GameForcing | PassForcing
agreedStrain{ type: none|suit|notrump, suit?, confidence? }
pendingActionNone | ShowMajor | ShowSuit
competitionModeUncontested | Doubled | Overcalled | Balancing
captainOpener | Responder | Neither
systemModeOn | Off | Modified (global default)
systemCapabilitiesRecord of capability → SystemMode (per-capability override)
conventionDataOpen record for convention-specific state
interferenceDetail{ call, seat, isNatural, kind? } if opponent interfered
framesStack of DialogueFrame for multi-step obligations

Algorithm: computeDialogueState(auction, conventionRules, baselineRules?)

state = INITIAL_DIALOGUE_STATE

for each entry in auction.entries (index i):

  if single-pass mode (no baselineRules):
    for each rule in conventionRules:     // linear scan
      if rule.matches(state, entry, auction, i):
        effect = rule.effects(state, entry, auction, i)
        state = applyEffect(state, effect)
        break                              // first match wins

  if two-pass mode (baselineRules provided):
    // Pass 1: convention rules (first-match-wins)
    alreadySet = empty set
    for each rule in conventionRules:
      if rule.matches(state, entry, auction, i):
        effect = rule.effects(...)
        alreadySet = getEffectKeys(effect)   // track which fields were set
        state = applyEffect(state, effect)
        break

    // Pass 2: baseline rules (first-match-wins, backfill only)
    for each rule in baselineRules:
      if rule.matches(state, entry, auction, i):
        effect = rule.effects(...)
        state = applyBackfillEffect(state, effect, alreadySet)
        break                // only sets fields NOT in alreadySet

return state

Two-pass mode separates convention-specific transitions from generic baseline transitions. Convention rules fire first and "claim" fields; baseline rules only backfill unclaimed fields. This prevents baseline rules from overriding convention-specific state while still providing sensible defaults.

TransitionRule interface: Each rule has id, matches(state, entry, auction, entryIndex), and effects(state, entry, auction, entryIndex). Effects use typed set* prefix fields (e.g., setFamilyId, mergeConventionData).

5. Effective Context: The Bundle

EffectiveConventionContext is the assembled bundle that downstream pipeline stages consume. It's built by buildEffectiveContext() and bundles everything needed for candidate generation.

Algorithm: buildEffectiveContext(raw, config, protocolResult, publicBelief?, lookupConvention?)

  1. Compute dialogue state via computeDialogueState() using config's transition rules (two-pass or single-pass depending on whether config.baselineRules exists).
  2. Classify interference: If dialogueState.interferenceDetail exists and opponent convention IDs are present, call classifyInterference() which may set InterferenceKind (natural-overcall, penalty-double, etc.).
  3. Filter active overlays: From config.overlays, keep only those where:
    • overlay.roundName === protocolResult.activeRound.name (must match the current round)
    • overlay.matches(dialogueState) returns true
  4. Sort overlays by priority (ascending -- lower number = higher precedence, default 0).
  5. Return bundle: { raw, config, protocolResult, dialogueState, activeOverlays, publicBelief }

6. Intent Collection: Gathering All Proposals

Intent collection traverses the hand subtree to gather all IntentNode leaves (not just the matched one). This produces the candidate pool that the selection algorithm chooses from. The algorithm is a recursive DFS over the hand portion of the tree.

Algorithm: collectIntentProposals(handTreeRoot, context)

// Step 1: Skip past auction conditions to find hand subtree root
root = findHandSubtreeRoot(handTreeRoot, context)
  // walks auction-condition decision nodes, following YES/NO based on
  // condition evaluation, until reaching a hand-condition node or leaf

// Step 2: Recursive DFS collecting ALL leaves
function collectLeaves(node, pathEntries, results):
  switch node.type:
    case "fallback":
      return                    // skip -- no intent here

    case "intent":
      results.push({
        intent: node.intent,
        nodeName: node.name,
        meaning: node.meaning,
        defaultCall: node.defaultCall,
        pathConditions: pathEntries,   // hand conditions on path to this leaf
        sourceNode: node,
      })
      return

    case "decision":
      // INVARIANT: must be a hand condition (auction conditions filtered above)
      if isAuctionCondition(node.condition): throw Error

      yesPath = [...pathEntries, { condition, requiredResult: true }]
      noPath  = [...pathEntries, { condition, requiredResult: false }]
      collectLeaves(node.yes, yesPath, results)   // recurse YES
      collectLeaves(node.no,  noPath,  results)   // recurse NO

results = []
collectLeaves(root, [], results)
return results

Key distinction from tree evaluation: Tree evaluation follows a single path (the matched path). Intent collection explores all branches -- every IntentNode in the hand subtree becomes a proposal. Each proposal carries pathConditions: the hand conditions that must be true/false to reach it. These conditions are used later to compute failedConditions (which conditions the actual hand doesn't satisfy).

7. Overlay System: Patches for Interference

Overlays modify the evaluation pipeline when unusual auction conditions arise (e.g., opponent interference). Each ConventionOverlayPatch has a matches(dialogueState) predicate and optional hooks. Overlays are activated during effective context building and applied during candidate generation.

Four overlay hooks (applied in strict order):

  1. replacementTree -- Full tree replacement. First overlay wins (only one can replace). Re-evaluates the tree from the new root.
  2. suppressIntent(intent, ctx) -- Remove proposals from the candidate pool. All overlays compose (each can suppress independently). Returns true to suppress.
  3. addIntents(ctx) -- Inject proposals not in the tree. All overlays concatenate. Injected intents have no sourceNode and are never considered "matched."
  4. overrideResolver(intent, ctx) -- Override the standard resolver for a proposal. First non-null result wins. Returns ResolverResult | null.

Activation: Overlays are filtered by roundName === activeRound.name AND matches(dialogueState). Sorted by priority (lower = higher precedence). All error handling is graceful: hook exceptions trigger onOverlayError callback but don't crash the pipeline.

Trigger overrides: Overlays can also carry triggerOverrides (Map of roundName → SemanticTrigger[]) which replace the protocol round's original triggers during protocol evaluation. These are collected by collectTriggerOverrides() before protocol dispatch runs.

8. Candidate Generation: Resolving Intents

generateCandidates() is the central orchestration function. It takes the raw proposals from intent collection, applies overlay hooks, resolves each proposal through its resolver, and produces ResolvedCandidate[].

Algorithm: generateCandidates(handTreeRoot, handResult, effectiveCtx)

  1. Step 1 -- Replacement tree: Apply first overlay's replacementTree if any. Re-evaluates tree from new root. Tags all proposals with provenance "replacement-tree" or "tree".
  2. Guard: If no IntentNode was matched after potential replacement, return empty candidates.
  3. Step 2 -- Collect proposals: collectIntentProposals(activeRoot, context) -- recursive DFS, all IntentNode leaves.
  4. Step 3 -- Suppress: Apply all overlays' suppressIntent hooks. Track whether the matched intent was specifically suppressed.
  5. Step 4 -- Add: Apply all overlays' addIntents hooks. Tags with provenance "overlay-injected".
  6. Step 5 -- Resolve each proposal (linear scan):
    1. Compute defaultCall by calling proposal.defaultCall(raw). If it throws, skip this proposal.
    2. Compute failedConditions: for non-matched proposals, re-evaluate each path condition against the raw context. Conditions that don't match their required result are failures.
    3. Check for overlay resolver override: scan overlays for first non-null overrideResolver result.
    4. If no override, call the standard resolver: resolveIntent(intent, dialogueState, raw, resolvers).
      • Returns ResolverResult: discriminated union of { status: "resolved", calls }, { status: "use_default" }, or { status: "declined" }.
      • resolved: Linear scan of calls array for first legal encoding. If none legal, use first call anyway.
      • use_default: Keep the defaultCall.
      • declined: Exclude this proposal entirely (returns null).
    5. Check legality: isLegalCall(auction, resolvedCall, seat).
    6. Build ResolvedCandidate with all metadata (bidName, nodeId, meaning, call, failedConditions, intent, source, resolvedCall, isDefaultCall, legal, isMatched, priority, provenance).
  7. Ordering: Matched candidates are unshifted to front; non-matched are pushed to back.

9. Candidate Selection: Tiered Algorithm

selectMatchedCandidate() picks the best candidate from the resolved pool using a strict four-tier priority system. Within each tier, the first candidate found in the (optionally ranked) array wins. The ranker influences which candidate wins within a tier, not tier boundaries.

Algorithm: selectMatchedCandidate(candidates, ranker?, forcingState?)

ranked = ranker ? ranker(candidates) : candidates

// Forcing constraints
noPass   = forcingState is ForcingOneRound OR GameForcing
passOnly = forcingState is PassForcing
allowed(c) = not (noPass AND c is Pass) AND not (passOnly AND c is not Pass)

// Tier 1: matched + legal + allowed
matched = ranked.find(c => c.isMatched AND c.legal AND allowed(c))
if matched: return matched

// Tier 2: preferred + legal + satisfiable + allowed
preferred = ranked.find(c =>
  c.priority === "preferred"
  AND c.legal
  AND c.failedConditions.length === 0    // satisfiable
  AND allowed(c))
if preferred: return preferred

// Tier 3: alternative + legal + satisfiable + allowed
alt = ranked.find(c =>
  c.priority === "alternative"
  AND c.legal
  AND c.failedConditions.length === 0
  AND allowed(c))
if alt: return alt

// Tier 4: no selection
return null

Tier 1

Matched + Legal

The tree's own answer

Tier 2

Preferred + Satisfiable

Convention-marked preference

Tier 3

Alternative + Satisfiable

Acceptable backup

Tier 4

null

No candidate viable

Forcing constraints: When ForcingOneRound or GameForcing is active, Pass is excluded from all tiers. When PassForcing is active, only Pass is allowed. These constraints are enforced at selection time and also by the strategy chain's resultFilter (belt-and-suspenders).

Satisfiability gate (Tiers 2-3): failedConditions.length === 0 -- the candidate's hand conditions must all be satisfied by the actual hand. This prevents selecting a "preferred 2H bid" when the hand doesn't have 4 hearts. Tier 1 (matched) doesn't need this check because the matched candidate is by construction the one whose path conditions all passed during tree evaluation.

10. Inference System: Building Beliefs from Bids

The inference system runs parallel to the evaluation pipeline. It doesn't determine what bid to make -- it determines what we know about each player's hand based on what they bid. This feeds into the practical scoring layer.

Key architecture decisions:

  • Two inference engines per deal: nsEngine (N/S perspective) and ewEngine (E/W perspective) -- models information asymmetry between partnerships.
  • Incremental: processBid() called after every bid, not at auction end.
  • Monotonic narrowing: Additional inferences can only tighten bounds, never widen them.
  • Range intersection: min = max(all minHcp), max = min(all maxHcp). Same per-suit.

Algorithm: mergeInferences(seat, inferences) -- Range Intersection

hcpMin = 0, hcpMax = 40
suitMins = { S:0, H:0, D:0, C:0 }
suitMaxes = { S:13, H:13, D:13, C:13 }

for each inference:
  hcpMin = max(hcpMin, inference.minHcp)
  hcpMax = min(hcpMax, inference.maxHcp)
  for each suit:
    suitMins[suit] = max(suitMins[suit], inference.minLength)
    suitMaxes[suit] = min(suitMaxes[suit], inference.maxLength)

// Contradiction clamping
if hcpMin > hcpMax:
  use last inference's values (most recent bid wins)
for each suit:
  if suitMin > suitMax: clamp suitMin = suitMax

Negative Inference

When a decision node's condition fails during tree evaluation (recorded in rejectedDecisions), the inverse tells us what the hand does NOT have. This is extracted by the tree inference system.

invertInference(ci) inversions:

OriginalInverted to
hcp-min: 12hcp-max: 11
hcp-max: 15hcp-min: 16
suit-min: 4 heartssuit-max: 3 hearts
suit-max: 3 spadessuit-min: 4 spades
hcp-rangedisjunction (resolved against cumulative state)

Uninvertible types (ace-count, king-count, two-suited) return null -- no false inference.

Private Belief Conditioning

conditionOnOwnHand() narrows partner beliefs using own hand knowledge. Partner HCP max is capped at 40 - ownHcp. Partner suit length max is capped at 13 - ownSuitLength. These are physical constraints (the deck has 40 total HCP and 13 cards per suit).

11. Practical Scoring: Weighted Evaluation

The practical scoring layer is advisory only -- it never changes the teaching answer or grade. It produces a PracticalRecommendation that tells the user what an experienced player might prefer. It scores candidates using beliefs (inference data) combined with hand evaluation.

Scoring Formula: scoreCandidatePractically()

// Weights
WEIGHTS = {
  fit:                    2.0,    // suit fit bonus
  hcp:                    1.5,    // HCP adequacy
  conventionDistance:     -4.0,   // penalty for non-matched candidates
  misunderstandingRisk:  -3.0,    // penalty for partner confusion
}

// Level-to-HCP floor table
LEVEL_HCP_TABLE = { 1:20, 2:23, 3:26, 4:26, 5:29, 6:33, 7:37 }

// For suit bids:
fitScore = ownSuitLength + partnerMinSuitLength
hcpScore = (ownHcp + partnerMinHcp) - LEVEL_HCP_TABLE[bidLevel]

// Convention distance:
//   matched candidate     -> 0
//   preferred candidate   -> 1
//   alternative candidate -> 2
//   pragmatic candidate   -> 3

totalScore =
    WEIGHTS.fit * fitScore
  + WEIGHTS.hcp * hcpScore
  + WEIGHTS.conventionDistance * conventionDistance
  + WEIGHTS.misunderstandingRisk * misunderstandingRisk

// Pass/double/redouble -> score 0 (all components zero)
// NT bids -> fitScore = 0 (no suit fit applies)

Candidate pool: The scorer evaluates both normative candidates (from the convention tree) and pragmatic candidates (heuristic tactical bids). Only legal + satisfiable normative candidates are scored. All candidates are sorted descending by score; the top scorer becomes the recommendation.

Pragmatic Candidate Generator

Three tactical heuristics produce bids that convention trees don't cover. Each is tried independently; results are deduplicated against existing convention candidates.

Conservative NT Downgrade

When partner bid NT and combined HCP is within 2 points of the level floor, suggest one level lower. Linear scan of auction entries backward to find partner's NT bid.

Competitive Overcall

When opponents have bid, 8+ HCP, and a 5+ card suit exists. Linear scan of 4 suits, for each finds cheapest legal bid (level 1-7 scan).

Protective Double

Passout seat protection: 10+ HCP, last 3 entries are opponent-bid + pass + pass, opponent at level 3 or below.

Partner Interpretation Model

computePartnerInterpretation() models what partner would infer from a candidate bid. It computes misunderstandingRisk (HCP deviation between what the bid promises and what the hand actually has) and continuationAwkwardness (suit length shortfall). The misunderstandingRisk feeds into the scoring formula as a penalty.

12. Teaching Resolution: Grading the User

Teaching resolution takes the BidResult and determines how to grade the user's bid. It's completely deterministic and isolated from the practical layer. Same hand + same auction = same grade, always.

Algorithm: resolveTeachingAnswer(bidResult)

  1. Primary bid = bidResult.call (the convention's answer).
  2. Extract resolvedCandidates from bidResult.treePath.
  3. If no candidates, return exact grading (only the primary bid is correct).
  4. Filter acceptable alternatives: candidates that are not matched, legal, failedConditions.length === 0 (satisfiable), and have priority "preferred" or "alternative".
  5. Compute ambiguity score: 0 preferred → 0.3, 1 preferred → 0.6, 2+ preferred → 0.8.
  6. Determine grading type: if matched candidate used resolver (non-default call) → "intent_based"; if acceptable alternatives exist → "primary_plus_acceptable"; else → "exact".

Algorithm: gradeBid(userCall, resolution)

if callsMatch(userCall, resolution.primaryBid):
  return BidGrade.Correct          // exact match to convention answer

if resolution.acceptableBids.some(bid => callsMatch(userCall, bid.call)):
  return BidGrade.Acceptable       // matched an acceptable alternative

return BidGrade.Incorrect          // neither primary nor acceptable

callsMatch() is a structural equality check on Call objects (same type, same level, same strain for bids). The three grades map to UI colors: green (Correct), teal (Acceptable), red (Incorrect). Acceptable bids auto-advance the auction.

13. Strategy Chain: Fallback Composition

The strategy chain is how multiple strategies compose. It wraps an ordered array of BiddingStrategy objects and tries each in sequence. First non-null result wins, subject to an optional resultFilter.

Algorithm: createStrategyChain(strategies, options?)

for each strategy in strategies:
  try:
    result = strategy.suggest(context)
    if result is not null:
      if resultFilter exists AND resultFilter(result, context) is false:
        trace.add(strategy.id, "filtered")   // rejected by filter
        continue                              // try next strategy
      trace.add(strategy.id, "suggested")
      return result (with merged trace)
    trace.add(strategy.id, "declined")        // returned null
  catch:
    trace.add(strategy.id, "error")           // strategy threw

return null                                   // all strategies exhausted

Production chain: [conventionStrategy, naturalFallbackStrategy] with createForcingFilter(config) as resultFilter. The forcing filter rejects Pass results when the dialogue state has active forcing. This provides an end-to-end guarantee: no strategy can produce Pass during a forcing auction.

Strategy chain in production (N/S partnership):

  1. Convention strategy (e.g., Stayman wrapped by conventionToStrategy) -- tries convention-specific logic
  2. Natural fallback -- 6+ HCP with a 5+ card suit → bid cheapest legal in that suit (linear scan: level 1-7, per suit by SUIT_ORDER)
  3. If all return null or are filtered: chain returns null → drill session wraps as Pass

14. Resolver Architecture: defaultCall vs resolvedCall

Every IntentNode carries two paths to a concrete bid. Understanding which is used where is essential to understanding the architecture.

defaultCall(ctx: BiddingContext) → Call

Always present on every IntentNode. Takes only the hand and auction context. Computes the concrete bid for the default (uncontested) context.

Used by: tree flattening, sibling finder, inference extraction, teaching display. Everything except the production decision path.

IntentResolverFn(intent, state, ctx) → ResolverResult

Optional. Looked up from IntentResolverMap by intent type. Takes DialogueState in addition to context. Can adapt the call based on forcing, interference, or system mode.

Used by: only generateCandidates() in the production strategy path.

ResolverResult: Three-Way Discriminated Union

type ResolverResult =
  | { status: "resolved", calls: ResolvedIntent[] }
      // Resolver provides one or more concrete calls.
      // Linear scan: first legal call wins.
      // If none legal, first call used anyway.

  | { status: "use_default" }
      // No special encoding needed.
      // Pipeline uses IntentNode.defaultCall() as-is.

  | { status: "declined" }
      // Intent is invalid in this context.
      // Candidate is EXCLUDED entirely from the pool.
      // Example: Stayman resolver returns declined when
      //   getSystemModeFor(state, STAYMAN_CAPABILITY) === Off

Convention variation: Conventions differ in how much they use resolvers:

  • SAYC: Empty resolver map (new Map()). All 55+ IntentNodes have deterministic defaultCall functions. Resolvers aren't needed because bids depend only on hand + auction, never on dialogue state.
  • Stayman: Full resolver map (9 entries). Each checks system mode and computes level dynamically (e.g., 2C after 1NT, 3C after 2NT).
  • Bergen: Partial map (5 entries). ShowShortage, AskShortage, HelpSuitGameTry use dynamic defaultCall functions because they need BiddingContext (which major was opened), not DialogueState.

The isDefaultCall flag: Set on each ResolvedCandidate. When true, the resolver didn't change the call. When false, the resolver remapped it. This flag flows through to ResolvedCandidateDTO and into the teaching system, which uses it to determine grading type ("intent_based" when a resolver produced a non-default call).

15. Three-Level Forcing Enforcement

Forcing state determines whether Pass is allowed. The system enforces forcing at three independent levels -- a belt-and-suspenders-and-belt approach that guarantees correctness.

Level 1: Candidate Selector

selectMatchedCandidate() excludes Pass candidates from all tiers when ForcingOneRound or GameForcing. Allows only Pass when PassForcing.

File: candidate-selector.ts:24-31

Level 2: Pragmatic Filter

conventionToStrategy() filters Pass from pragmatic candidates when forcing is active. Prevents tactical pass suggestions during forcing auctions.

File: convention-strategy.ts:185-188

Level 3: Strategy Chain Filter

createForcingFilter(config) on the chain's resultFilter rejects Pass results from ANY strategy. End-to-end guarantee: even passStrategy can't produce Pass during forcing.

File: config-factory.ts + strategy-chain.ts:24-26

Forcing states are set by DialogueState.forcingState, computed via transition rules during auction replay. For example, Stayman's 2C response sets ForcingOneRound -- opener must respond to the Stayman ask.

Trace diagnostics: forcingFiltered flags when the chain rejected a strategy result due to the forcing filter.

17. Store Integration: How Evaluation is Triggered

The store layer connects the evaluation pipeline to the UI. Understanding how userBid() triggers evaluation reveals the full lifecycle of a bid decision.

Algorithm: bidding.svelte.ts :: userBid(call)

  1. Guard checks: Must have deal, session, not be processing, must be user's turn.
  2. Get expected result: If conventionStrategy exists, call conventionStrategy.suggest(context) to get what the convention says is correct. This triggers the entire pipeline (sections 1-13).
  3. Resolve teaching answer: resolveTeachingAnswer(expectedResult) -- extracts acceptable alternatives from BidResult.treePath.resolvedCandidates.
  4. Grade the bid: gradeBid(userCall, teachingResolution) -- Correct, Acceptable, or Incorrect.
  5. Build feedback: BidFeedback { grade, userCall, expectedResult, teachingResolution, practicalRecommendation }.
  6. Apply call to auction: engine.addCall(auction, call, seat).
  7. Feed inference engines: onProcessBid callback feeds both NS and EW inference engines + produces public belief annotation via produceAnnotation() → applyAnnotation().
  8. Record in history: Push BidHistoryEntry with isUser: true, isCorrect, and expectedResult (if wrong).
  9. Branch: If correct (Correct or Acceptable) → check auction completion, run AI bids. If incorrect → show feedback panel (user can dismiss, retry, or skip).

AI bid loop (runAiBids()): While the current turn is not the user's, wait 300ms, call session.getNextBid(seat, hand, auction), apply the call, feed inference engines, and advance. If the strategy returns null, wraps as Pass.

Drill session wiring: createDrillConfig() in drill/config-factory.ts builds the strategy chain per seat. N/S seats get [conventionStrategy, naturalFallbackStrategy] with a forcing filter. E/W seats get the opponent convention (default SAYC) or passStrategy. The user's seat gets "user" (returns null, signals UI to wait).

Layer Classification

LayerTypeWhat it decides
Protocol Dispatch Deterministic Which round of the convention applies, which seat acts
Tree Evaluation Deterministic Which IntentNode the hand matches in the convention tree
Dialogue State Deterministic What the auction means (forcing, agreed strain, system mode)
Intent Collection Deterministic All possible bids and their hand conditions
Overlay Hooks Deterministic How interference modifies the candidate pool
Candidate Selection Deterministic Which single bid to make (4-tier priority)
Inference Engine Non-deterministic* What we believe about each player's hand (range narrowing)
Practical Scorer Non-deterministic* What an experienced player might prefer (advisory)
Teaching Resolution Teaching Whether the user's bid is correct, acceptable, or incorrect

* These layers are deterministic in implementation (same inputs = same outputs), but operate on probabilistic/belief data (inferred ranges), hence "non-deterministic" in the semantic sense.
