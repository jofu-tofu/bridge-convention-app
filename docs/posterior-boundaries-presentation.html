Posterior feedback, translated into one question: what belongs where?

This presentation turns the posterior-engine feedback into a boundary map. It shows what the feedback changes, what it leaves alone, and how it fits the current contracts and files in this codebase.

Main takeaway

Deterministic teaching stays upstream. Probabilistic realism becomes a separate full-world posterior layer.

Canonical state

The engine should reason over weighted particles of complete hidden worlds, not seat-local marginals.

Big rename

publicBeliefs should leave PublicSnapshot. Snapshot carries public state, not beliefs.

Big implementation shift

Move from rejection sampling plus fact handlers toward full-world importance sampling plus compiled query IR.

Executive Read

The feedback does not move correctness into probability. That boundary stays exactly where the current specs want it: the meaning pipeline and teaching stack still decide what is correct, acceptable, and recommended. The posterior layer stays a realism layer that explains uncertainty and plausibility.

What changes is the canonical probabilistic object. Today, the code and contracts lean toward PublicHandSpace, SeatPosterior, and snapshot-level publicBeliefs. The feedback says those are the wrong center of gravity. The center should be a joint distribution over full hidden worlds: unseen cards, unresolved branch choices, and optional latent variables.

In plain English: the current model mostly asks, “what might this one seat hold?” The feedback says the canonical model should ask, “what complete bridge worlds are still alive, and how much weight should each one have?” Seat beliefs then become projections.

Boundary Rules

The most useful part of the feedback is that it turns several fuzzy ideas into a small set of yes-or-no classification rules. If you remember only one section, remember this one.

Rule 1: Impossible vs unlikely

This is the hard/soft boundary.

  • If violating it makes a world impossible, it belongs in ConstraintProgram.
  • If violating it only lowers plausibility, it belongs in EvidenceProgram.
  • If it is baseline mass before observed realism, it belongs in PriorProgram.

Rule 2: Public schema vs latent realization

This is the ambiguity boundary.

  • publicAmbiguitySchema is public and deterministic.
  • BranchAssignment is per-world and hidden.
  • Do not collapse those into one mixed object.

Rule 3: Canonical state vs read model

This is the posterior object boundary.

  • The canonical probabilistic object is a distribution over LatentWorld.
  • SeatPosteriorView, branch summaries, and fact values are projections.
  • PublicHandSpace is a support aid, not the canonical belief object.

Rule 4: Deterministic teaching vs probabilistic realism

This is the biggest architectural guardrail.

  • The meaning pipeline still defines correctness.
  • The posterior engine may explain realism and uncertainty.
  • The posterior engine must never redefine the answer key.

Practical litmus test

When you are unsure where something belongs, ask three questions in order: Is it public? Is it deterministic? Does it make worlds impossible or just more or less plausible? Those three answers usually place it in the right subsystem immediately.

How It Fits the Current Code

The good news is that this feedback fits your current architecture direction better than it fits the current posterior implementation. In other words, it mostly reinforces the spec philosophy and asks the posterior code to catch up.

Area Already aligned Boundary mismatch the feedback exposes
Teaching architecture Root project context already says deterministic teaching and probabilistic realism are separate. Snapshot-level publicBeliefs blurs that separation by embedding posterior summaries into public state.
Public commitments publicRecord and publicCommitments already exist and are the right upstream inputs. Posterior compilation currently turns those mostly into per-seat hand predicates, which is narrower than the proposed full-world model.
Ambiguity The code already has LatentBranchSet as a concept. The type still mixes schema-ish and probability-ish concerns; the feedback separates public schema from realized hidden selection.
Posterior facts There is already a contract boundary for posterior facts and a catalog bridge into the fact system. The current implementation is still handler-per-fact, not compiled queries over a general algebra.
Sampling Monte Carlo is already the instinct, and the system already has a sampler seam. Current samples are accepted with unit weight, so support and realism are not cleanly separated.

Files that likely change semantics

  • src/core/contracts/module-surface.ts — remove posterior beliefs from snapshot semantics.
  • src/core/contracts/posterior.ts — redefine canonical objects, views, evidence programs, and query IR.
  • src/core/contracts/fact-catalog.ts — shift posterior evaluators toward query-template compilation.

Files that likely change implementation shape

  • src/inference/posterior/posterior-engine.ts — build PosteriorContext and PosteriorState instead of seat-first objects.
  • src/inference/posterior/posterior-sampler.ts — move from rejection sampling to weighted full-world proposals.
  • src/inference/posterior/posterior-facts.ts — retire fact-level handlers as the default extension pattern.

Questions This Feedback Resolves

Open question Recommended resolution Boundary effect
Canonical latent state Use a distribution over full hidden worlds, including branch assignments. Joint facts become native. Seat marginals become derived.
Hard vs soft vs prior Use three explicit programs: ConstraintProgram, PriorProgram, EvidenceProgram. Support, baseline mass, and realism stop leaking into each other.
Correlation semantics Use typed evidence groups instead of prose-only likelihood assumptions. Engine can tell when to multiply, sum alternatives, score jointly, or score sequentially.
Posterior facts Compile fact requests through query templates into PosteriorQueryIR. Fact catalog owns templates; primitive operators own hard logic.
Witness relationship Share a lower-level world predicate substrate, but keep top-level witness and posterior IRs separate. Deterministic teaching search and probabilistic inference stay distinct.
Belief placement Beliefs leave PublicSnapshot; views come from posterior runtime state. Public state becomes a cleaner machine-to-posterior seam.

Migration Path in Plain English

The migration path implied by the feedback is clean because it mostly redraws contracts before it demands algorithmic sophistication. You do not need to jump to full sequential Monte Carlo on day one.

  1. 1. Clean up semantics first. Reclassify current inputs into public snapshot, hard constraints, priors, evidence, and ambiguity schema.
  2. 2. Reset the canonical object. Make full hidden worlds the center; demote hand spaces and seat posteriors to support and view layers.
  3. 3. Add weights before sophistication. Replace unit-weight rejection sampling with weighted importance sampling over full worlds.
  4. 4. Replace vague likelihoods with typed evidence groups. This resolves the evidence-correlation open question structurally.
  5. 5. Replace fact handlers with query templates. That resolves the fact-catalog compiler question at the right boundary.
  6. 6. Extract the shared world substrate. Witness generation and posterior inference reuse the same lower-level predicates without becoming the same IR.
  7. 7. Upgrade the runtime later. Once semantics are clean, move from snapshot-wise importance sampling to sequential Monte Carlo with resampling and conditioned decision views.

Bottom Line

The feedback is best understood as a boundary correction, not a new theory of the whole app. It keeps your existing architecture story intact: public runtime upstream, meaning pipeline for correctness, teaching projection for pedagogy.

The big correction is local to inference: stop treating seat-local hand spaces and snapshot-level belief views as the main probabilistic object. Instead, treat them as helpers and projections around one canonical posterior state over full hidden bridge worlds.

If you adopt that center of gravity, several current open questions become easier to answer consistently: publicBeliefs naming, evidence correlation semantics, posterior fact compilation, and the boundary with WitnessSpecIR all become cleaner because each now has one obvious home.

One-sentence memory aid

The meaning pipeline decides what is correct; the posterior engine decides which complete hidden worlds are still plausible.
