Understanding the src/ folder without getting lost in code jargon

This is a plain-language map of what each part of src/ owns, how they talk to each other, and where the code is tightly connected versus nicely separated.

Overall read

Mostly layered. The folder is not a tangled ball of code.

stores/ is the busiest hub. That is where most pieces meet.

Big Picture

The src/ folder is split by job. One part knows the game rules. One part knows bidding systems. One part chooses moves. One part keeps app state. One part draws the screens. That is a good sign: the project is trying to keep “what something means” separate from “how it is shown”.

The deepest coupling is not spread everywhere. It is concentrated in a few places on purpose, mostly in stores/, which acts like mission control. The lower-level folders such as engine/, display/, and util/ are much more self-contained.

Folder Guide

engine/

The rules referee. It knows what is legal in bridge: legal calls, contract scoring, trick winners, deal generation, and similar core rules.

conventions/

The bidding-system library. It knows what Stayman, Bergen Raises, SAYC, and the other teaching systems mean.

strategy/

The move picker. It uses the rules, the convention knowledge, and some inference to decide what the AI should do.

inference/

The interpretation layer. It tries to read what bids suggest about unseen hands.

drill/

The teaching-session layer. It starts drills, sets up seats and deals, and grades the user’s answer.

stores/

The app traffic controller. It keeps the current state of the game, decides which phase the app is in, and connects the UI to the deeper logic.

components/

The visible app. These are the screens, panels, buttons, tables, and card views.

display/

The presentation helper shelf. It formats bids, maps seats, sorts cards, and prepares teaching information for the screen.

shared/

The common language. It holds shared data shapes used by more than one area.

util/ and test-support/

Utility drawers. Small helpers live in util/. Test-only helpers live in test-support/.

How They Interact

Most of the interaction is routed through the stores. A screen does not usually reach deep into the codebase by itself. Instead, the screen talks to the store, and the store reaches into the engine, strategy, drill, or display layers.

That means the project is not flat. It has a center of gravity. If you want to trace “what happens when I click a button?”, start with components/, then go to stores/, then follow the store into the deeper layers.

Plain-English takeaway

The codebase is not “everything talks to everything.” It is closer to “many things report to the store layer, and the store layer coordinates the rest.”

Where Coupling Is Strong

1. stores/ is intentionally coupled to many areas

This is the strongest coupling point in the folder. That is not automatically bad. It is acting as the app coordinator, so it needs to know about engine calls, drill sessions, AI decisions, screen state, and phase changes.

2. strategy/ sits in the middle of several idea systems

Strategy depends on convention rules, bridge rules, and inference. That makes it a “thinking layer,” so some coupling is expected there too.

3. engine/ is comparatively clean

The bridge rules engine is deliberately kept away from Svelte, browser details, and convention-teaching logic. That is one of the cleanest boundaries in the project.

4. display/ and util/ are low-risk helpers

These folders mostly help other layers present or compute things. They are supportive, not controlling.

Coupling Scorecard

Folder What it mainly does Coupling level Why
components/ Shows the app Low to medium Mostly leans on stores instead of reaching everywhere
stores/ Coordinates the app High It is the meeting point for state, flow, and deeper logic
drill/ Runs teaching sessions Medium Needs several systems, but in a structured teaching role
strategy/ Chooses AI actions Medium to high It combines rules, conventions, and interpretation
conventions/ Stores bidding-system knowledge Medium It is central to teaching, but kept separate from UI and app state
inference/ Interprets what bids imply Medium Touches strategy and conventions, but stays out of the UI
engine/ Knows bridge rules Low Cleanly separated from UI and teaching logic
display/ Formats things for the screen Low Helpful support layer, not a decision-making hub
shared/ Shared data language Low Used across boundaries to reduce deeper entanglement

Bottom Line

The src/ folder has some real complexity, but it is not deep coupling everywhere. The code is mostly organized around a sensible split: rules, teaching knowledge, decision-making, state coordination, and UI.

If you are asking “where would changes ripple the most?”, the answer is stores/ first, strategy/ second, and then parts of conventions/. If you are asking “which areas are safest and most self-contained?”, the answer is engine/, display/, util/, and much of shared/.

So the honest read is: moderately coupled at the orchestration layer, fairly well-separated underneath.
