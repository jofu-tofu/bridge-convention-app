System Overview

The bridge-convention-app teaches bridge bidding conventions through interactive drills. Its TypeScript frontend is organized into 10 distinct modules with an intended layered architecture: a pure game engine at the bottom, convention definitions and evaluation in the middle, and strategy/inference/drill/display/stores at the top driving the Svelte 5 UI.

Key finding: The conventions/core/ module is the gravitational center of the codebase (38 files, 5,206 LOC) with high fan-in and fan-out. A bidirectional type dependency between shared/ and conventions/core/ creates an architectural cycle. The drill/config-factory.ts file is a wiring mega-hub that imports from 5 different modules.

Module Map

The intended architecture is a layered DAG (directed acyclic graph). Lower layers know nothing about upper layers. The engine is the pure foundation; shared types sit at the boundary; conventions, inference, and strategy form the domain logic; drill/display/stores wire everything together for the UI.

Observations from the Matrix

  • conventions/core/ has 36 imports from engine/ — the heaviest single coupling edge in the codebase
  • shared/ and conventions/core/ are mutually dependent — the only circular dependency (marked with ⚠)
  • drill/ touches 5 modules — highest fan-out of any non-UI module
  • strategy/bidding/ has 20 imports from conventions/core/ — deep structural coupling, not just type imports
  • display/ has 18 imports from conventions/core/ — a surprising amount for a "display utility" module
  • engine/ is clean — zero external imports (as intended)
  • util/ is clean — zero external imports

Why conventions/core/ is the gravitational center

  • 6 modules import from it: strategy/, inference/, drill/, display/, stores/, shared/ (the most fan-in of any module)
  • 38 files with deep internal coupling: The candidate pipeline alone involves 6 files (intent-collector, candidate-generator, candidate-selector, candidate-builder, effective-context, interference-classifier)
  • Consumers import from 8+ different core files: strategy/bidding/convention-strategy.ts imports from registry, effective-context, candidate-generator, candidate-selector. tree-eval-mapper.ts imports from types, rule-tree, tree-evaluator, sibling-finder, candidate-generator.
  • Types defined deep inside core/ leak outward: BeliefData from effective-context.ts, ResolvedCandidate from candidate-generator.ts, TreeEvalResult from tree-evaluator.ts are all consumed by external modules

Why it exists: BiddingStrategy.suggest() takes a BiddingContext parameter (defined in conventions/core/types.ts). Meanwhile, conventions/core/ needs ForcingState, CandidateBid, and SiblingBid from shared/types.ts for its pipeline output.

Impact: These modules cannot be loaded independently. Changing either requires awareness of the other. TypeScript's import type makes this safe at runtime (no circular initialization), but it couples the type systems.

Strategy

Strategy wraps convention evaluation into BiddingStrategy and PlayStrategy interfaces. The convention-strategy.ts file is the primary adapter between conventions/core/ and the rest of the system.

strategy/bidding/ (8 files)

  • convention-strategy.ts — conventionToStrategy(config, options?) — the mega-adapter
  • strategy-chain.ts — createStrategyChain() tries strategies in order
  • tree-eval-mapper.ts — maps TreeEvalResult to TreeEvalSummary DTO
  • practical-scorer.ts — pure scoring functions
  • practical-types.ts — local types for practical pipeline
  • pragmatic-generator.ts — tactical bids (NT downgrade, overcall, etc.)
  • natural-fallback.ts — fallback strategy for natural bids
  • pass-strategy.ts — always-pass placeholder
  • fit-ranker.ts — candidate ranking by fit confidence

convention-strategy.ts imports

This single file imports from 5 modules:

  • conventions/core/types — ConventionConfig, BiddingContext, ConventionLookup
  • conventions/core/registry — evaluateBiddingRules
  • conventions/core/effective-context — buildEffectiveContext, BeliefData
  • conventions/core/candidate-generator — generateCandidates, ResolvedCandidate
  • conventions/core/candidate-selector — selectMatchedCandidate
  • shared/types — ForcingState, BidResult, etc.
  • shared/hand-summary — formatHandSummary
  • engine/constants — partnerSeat, SUIT_ORDER

Coupling concern: convention-strategy.ts reaches into 5 different files within conventions/core/ (types, registry, effective-context, candidate-generator, candidate-selector). This is not just importing types — it imports runtime functions like evaluateBiddingRules(), buildEffectiveContext(), generateCandidates(), and selectMatchedCandidate(). Any refactor of the candidate pipeline requires updating this file.

Coupling concern: config-factory.ts has 20 import lines pulling from 5 different modules. Analysis

7 out of 13 display/ files import from conventions/core/. Several import runtime functions, not just types:

  • evaluateConditions() from condition-evaluator.ts
  • buildExplanation() from condition-evaluator.ts
  • createBiddingContext() from context-factory.ts
  • getEffectiveRules() from registry.ts
  • evaluateAllBiddingRules() from registry.ts
  • reconstructBiddingContext() from debug-utils.ts

This means display/ is not a pure formatting layer — it actively calls into the convention evaluation pipeline. This is a significant coupling concern for a module named "display."

dds.svelte.ts engine/ (port, types) — minimal coupling
app.svelte.ts conventions/core/ (ConventionConfig) — type-only

Observation: game.svelte.ts has the heaviest import footprint, pulling from 5+ modules including 7 different files from inference/. This is where the inference engine lifecycle is managed: creating engines, processing bids through annotation producers, maintaining belief state. Notably, stores/ does not import deeply from conventions/core/ — only createBiddingContext and ConventionConfig (type). The heavy convention wiring is delegated to drill/.

Coupling Concerns

Ranked by severity. Each concern is an observation, not a prescription.

Critical

shared/ and conventions/core/ circular dependency

shared/types.ts imports BiddingContext from conventions/core/types.ts. conventions/core/ imports ForcingState, CandidateBid, SiblingBid, SiblingConditionDetail, BidAlert from shared/types.ts.

Root cause: BiddingStrategy.suggest() in shared/ takes a BiddingContext parameter, which is defined in conventions/core/. Meanwhile, the candidate pipeline in core/ needs shared DTO types for its output.

Blast radius: Neither module can be understood, tested, or loaded in isolation. Any type change in either module requires checking the other.

Warning

conventions/core/ is too large and has too many external consumers

38 files, 5,206 LOC, consumed by 6 modules. Internal subsystems (candidate pipeline, overlay system, dialogue system, intent system) each have 4-7 files with tight internal coupling. External consumers import from 8+ different files within core/.

Symptom: Changing any part of the candidate pipeline (e.g., how generateCandidates() works) requires checking strategy/bidding/convention-strategy.ts, tree-eval-mapper.ts, and potentially inference/convention-inference.ts. The module has no clear internal boundaries separating its subsystems.

Warning

strategy/bidding/ reaches deep into conventions/core/ internals

convention-strategy.ts imports from 5 different core/ files and calls 4 runtime functions (evaluateBiddingRules, buildEffectiveContext, generateCandidates, selectMatchedCandidate). tree-eval-mapper.ts imports from 4 core/ files.

Symptom: Strategy is not consuming a stable interface — it's orchestrating the candidate pipeline step-by-step. If the pipeline's internal structure changes, strategy must change too.

Warning

display/ imports runtime functions from conventions/core/

7 of 13 display files import from conventions/core/. Several import runtime functions (evaluateConditions, buildExplanation, createBiddingContext, getEffectiveRules, evaluateAllBiddingRules, reconstructBiddingContext).

Symptom: A module called "display" should format data, not evaluate conventions. Files like rules-display.ts and debug-bid-eval.ts are doing convention evaluation work that belongs elsewhere.

Info

drill/config-factory.ts is a mega-hub

20 import lines from 5 modules (engine, shared, conventions/core, strategy, inference). This is a composition root — acceptable in principle, but concentrated in a module named "drill" rather than a dedicated wiring/bootstrap module.

Trade-off: Composition roots are necessary. The question is whether drill/ is the right home for this responsibility, or whether it should be elevated to a top-level wiring module that drill/, stores/, and the app bootstrap all consume.

Info

Deep-module types leak across boundaries

Types like BeliefData (effective-context.ts), ResolvedCandidate (candidate-generator.ts), TreeEvalResult (tree-evaluator.ts), and ProtocolEvalResult (protocol.ts) are defined deep inside conventions/core/ but imported by strategy/ and inference/.

Symptom: External modules need knowledge of core/'s internal file structure. If files are renamed or types move, multiple external modules break. Some of these should be elevated to the core/ public surface (index.ts or a dedicated API types file).

OK

engine/ purity is well-maintained

Engine has zero runtime imports from any other app module. The one import type in bid-suggester.ts is erased at runtime and documented. Transport adapters properly isolated.

OK

No upward dependencies from lower layers

engine/ never imports from strategy/, inference/, drill/, display/, stores/, or components/. conventions/core/ never imports from strategy/ or inference/. The layered architecture holds — dependencies flow downward with the notable exception of the shared/ ↔ conventions/core/ cycle.

Duplicated Code Across Modules

The subagent analysis found several cases of duplicated logic across module boundaries. These are not just DRY violations — they represent hidden coupling where the same concept is maintained in multiple places.

1. LEVEL_HCP_TABLE duplicated in strategy/

The HCP floor per bid level table appears in both practical-scorer.ts (line 10-12) and pragmatic-generator.ts (line 26-28) with identical values.

Should be a shared constant in strategy/bidding/ or engine/constants.ts.

2. callKey() / callKeyForDedup() duplicated

pragmatic-generator.ts has a local callKey() function (lines 39-47) that is functionally identical to callKeyForDedup() exported from tree-eval-mapper.ts (lines 194-201).

The pragmatic generator builds an existingCalls set but uses its own key function instead of the exported one.

3. mergeToCumulative logic duplicated in inference/

Nearly identical cumulative merge logic appears in both protocol-inference-extractor.ts (lines 71-94) and tree-inference-extractor.ts (lines 66-89).

Could be extracted to a shared helper in inference/.

4. delay function duplicated in stores/

play.svelte.ts imports delay from util/delay, but bidding.svelte.ts defines its own inline defaultDelay function. They are functionally identical.

Minor inconsistency — both should use util/delay.

5. PrivateBeliefState construction duplicated

convention-strategy.ts constructs PrivateBeliefState inline (lines 151-167) rather than calling conditionOnOwnHand() from inference/private-belief.ts, duplicating the narrowing logic.

This means changes to belief narrowing must be made in two places.

Refactoring Observations

Based on the coupling analysis, here are architectural observations that could inform a refactoring effort. These are observations, not recommendations — the right approach depends on your priorities and constraints.

1. Break the shared/ ↔ conventions/core/ cycle

The cycle exists because BiddingStrategy.suggest() in shared/ takes BiddingContext (defined in core/) as a parameter. Options:

  • Move BiddingContext to shared/ — it's really a cross-boundary type. Would require moving or re-exporting its dependencies too.
  • Make BiddingStrategy.suggest() generic — use a type parameter or a simpler context interface defined in shared/.
  • Introduce a contracts/ or api/ module — a new module that owns all cross-boundary interfaces, replacing shared/.

2. Give conventions/core/ a public API surface

External modules import from 8+ different files within core/. This means they depend on core/'s internal file structure. A facade pattern could help:

  • Define explicit public APIs for each external consumer: core/api/strategy-api.ts, core/api/inference-api.ts (already partially exists), core/api/display-api.ts
  • Have external modules import only from the API files, never from internal modules like candidate-generator.ts
  • The candidate pipeline orchestration currently done in convention-strategy.ts could move inside core/ behind a facade

3. Split conventions/core/ into subsystems

At 38 files, core/ contains at least 5 distinct subsystems that could be separate modules:

Tree system: rule-tree, tree-evaluator, tree-compat
Protocol system: protocol, protocol-evaluator
Candidate pipeline: intent-collector, candidate-generator, candidate-selector, candidate-builder, effective-context
Dialogue system: dialogue/, event-classifier
Overlay system: overlay, overlay-tree-replacement

4. Reclassify display/ files that do evaluation

Files like rules-display.ts, debug-bid-eval.ts, and teaching-content.ts call into the convention evaluation pipeline. They could be:

  • Moved to a teaching/ or evaluation/ module that explicitly depends on conventions/core/
  • Refactored to accept pre-computed data instead of calling evaluation functions directly
  • The pure formatting utilities (format.ts, tokens.ts, sort-cards.ts, seat-mapping.ts, hcp.ts, table-scale.ts) would remain in display/

5. Consider a dedicated wiring/bootstrap module

drill/config-factory.ts is a composition root that imports from 5 modules. If wiring logic also exists in stores/ (inference engine lifecycle in game.svelte.ts), extracting a wiring/ or bootstrap/ module could centralize all dependency assembly and reduce drill/'s fan-out from 5 to 2-3.

6. Consolidate duplicated logic

Five instances of duplicated code were found across module boundaries (see Duplicated Code section above). The most impactful are the PrivateBeliefState construction duplication between strategy/ and inference/ (logic divergence risk) and the mergeToCumulative duplication within inference/ (easy fix). The LEVEL_HCP_TABLE and callKey duplications within strategy/ are low-risk but easy wins.

7. Overall architecture health

Despite the coupling concerns, the architecture is fundamentally sound:

  • No runtime circular dependencies — all cycles are type-only, erased by TypeScript
  • Dependencies flow downward — engine/ is never imported by upper layers except through types
  • Convention definitions are self-contained — no cross-convention imports
  • Play strategies are fully isolated — only depend on engine/ and shared/
  • Components never imported by non-UI code — UI boundary is clean
  • DI patterns (ConventionLookup, EnginePort, factory functions) enable testability

The main risk is gravitational pull toward conventions/core/ — as the module grows, more external modules depend on its internals, making future changes increasingly expensive. A public API surface and internal modularization would address this proactively.
