Backend Module Architecture

Bridge Convention App — Inter-module I/O and boundary types

Overview

Think of it as a compiler pipeline. Raw cards and auction state enter at the bottom (engine), get parsed into convention meanings (conventions), optimized into decisions (strategy + inference), assembled into runnable sessions (bootstrap), and exposed through a single async port (service) that UI and CLI consume.

Seven backend modules, one hexagonal boundary. The service/ module is the only door between domain logic and the outside world — identical to a database adapter in web frameworks. Everything behind that door is pure TypeScript with zero platform dependencies.

engine/ — Pure domain primitives

Imports: nothing. This is the leaf node — pure bridge rules with zero convention or platform knowledge. Every other module depends on engine types as the lingua franca.

Key exports consumed by other modules

Enums: Seat, Suit, Rank, BidSuit, Vulnerability
Domain types: Call, Card, Hand, Deal, Auction, AuctionEntry
Game state: Contract, Trick, PlayedCard, DDSolution

Functions: evaluateHand(), calculateHcp(), callKey()
Port: EnginePort (Rust/WASM interface)

conventions/ — Meaning pipeline

Four sub-modules behind a single barrel (conventions/index.ts). External consumers never import sub-modules directly. Internally, it works like a 4-stage compiler: core (grammar) → definitions (source code) → pipeline (compilation) → teaching (error messages).

inference/ — Belief engine

Tracks what each player's bids reveal about their hand. Two layers: a natural inference provider (rule-based: 6+ HCP, 5+ suit) and a posterior engine (factor graphs + particle sampling). The posterior sub-module compiles FactConstraints into a factor graph, then samples deals consistent with observed bidding.

I/O boundaries

Imports from engine/: Seat, Hand, Card, Auction, AuctionEntry, Call, Vulnerability + seeded-rng for sampling

Imports from conventions/core/: FactConstraint, FactCatalog, HandPredicate

Imports from strategy/bidding/: BidResult, BidAlert (for annotation production)

Exports to bootstrap/service: InferenceConfig, InferenceCoordinator, PublicBeliefState, InferenceSnapshot

Two sub-modules: bidding/ (convention-driven bid selection via pipeline) and play/ (heuristic card play). Bidding strategies are composable via createStrategyChain() — meaning-strategy first, natural-fallback second, pass-strategy as terminal.

bootstrap/ — Dependency assembly

The "linker" — wires engine, conventions, strategy, and inference into a runnable drill session. startDrill() is the single entry point: resolves the convention bundle, builds strategies for all four seats, configures inference engines, generates a constrained deal, and returns a DrillBundle ready for service/ to drive.

I/O boundaries

Imports from engine/: Seat, Deal, Hand, Auction, Vulnerability + deal-generator + seeded-rng

Imports from conventions/: createBiddingContext(), getBundle(), specFromBundle(), resolveConventionForSystem()

Imports from strategy/: BiddingStrategy, BidResult, protocolSpecToStrategy(), createHeuristicPlayStrategy()

Imports from inference/: InferenceConfig, createInferenceEngine()

Exports to service: DrillBundle, DrillSession, DrillConfig, DrillTuning

service/ — Hexagonal port

The sole interface between domain logic and consumers. UI (stores/components) and CLI both import exclusively from service/. Internally, it manages session state, drives the phase machine (bidding → declarer-prompt → playing → explanation), and builds sanitized viewport DTOs that expose only what the player should see.

I/O boundaries

Imports from: All 5 backend modules (engine, conventions, inference, strategy, bootstrap)

Exports to UI/CLI: ServicePort, SessionHandle, viewport types, feedback types, convention info

cli/ — Headless evaluation

Consumes service/ exclusively. Commands: eval, play, plan, selftest, verify explore|motif|fuzz|preflight. Uses the same ServicePort as the UI — no backdoor imports into backend modules.

Architectural Constraints

ESLint-enforced rules that maintain these boundaries at CI time.

ENFORCED
Hexagonal port: UI (components/ + stores/) imports only from service/. No direct backend imports.
ENFORCED
Deep import ban: External consumers cannot import from conventions/core/, conventions/pipeline/, etc. — only via conventions/index.ts.
ENFORCED
Engine purity: engine/ imports nothing from Svelte, Tauri, DOM APIs, or strategy/.
CONVENTION
No circular imports: Teaching files use direct sibling imports (e.g., ../pipeline/pipeline-types), never the conventions barrel.
CONVENTION
Backend free imports: engine/, conventions/, inference/, strategy/, bootstrap/ can import from each other freely (no inter-module restriction within the backend).
